# 对账差异闭合(Conformance Gap Closure)

## 用途

输入一份"文档处理管线代码 vs 开发文档"的**差异对账表**,把其中真正该补的差异闭合掉。这件事的核心不是"多写代码",而是**精确区分"该补的"和"不该碰的",并且只有测试通过才算闭合**。代码结构补上不等于行为正确——所以本 skill 的每一步都以测试为最终裁判,而不是以"我实现完了"为准。

本 skill 假设对账表是上游差异分析的产物,每行形如:

```
文档项 | 代码状态(在/漂移/缺) | 分类(完成/真缺/触发驱动) | 动作 | 触发条件(若属删)
```

例:`S0.5 LibreOffice 渲染 | 在 | 完成 | — | —`、`Temporal 编排 | 缺 | 触发驱动 | 不补 | 信创内网可部署性验证通过`、`chunk_id 公式 | 漂移 | 真缺 | 修正至契约 | —`。

---

## 四条硬约束(贯穿全程,违反即停)

这四条和迁移阶段一脉相承,**任何一项要被违反时,停下来报告,不要自行变通**:

1. **契约不动。** IR schema 字段、`chunk_id` 公式(`sha1(doc_version_id + "|" + clause_path_norm + "|" + seq)[:24]`)、PG 表字段名、Milvus collection schema、manifest 列——**只许让代码去符合契约,绝不许改契约值**。契约的权威来源是 `libs/common` 与设计文档(§4.1 / §4.2 / §6.5 / §10),不是某一处代码现状。

2. **补缺,不重写。** 这是逐项闭合差异,不是重构。能跑的代码不要"顺手优化"。补全一项后,该项之外的行为必须和补全前完全一致。

3. **不碰范围外。** "完成"和"触发驱动"两类行**不实现、不改动**,最多只读核对。范围外的过度热心是本任务最常见、最危险的失败模式。

4. **不确定就停下问,绝不猜。** 尤其是契约值的"权威之争"(文档说 A、代码说 B,谁对?)——这不是你能裁的,列出来交人决定。

---

## 输入与前置

调用时给出对账表路径($ARGUMENTS)。开工前读三样东西建立基准:

- **对账表**本身——这是本次的任务清单与账本。
- **契约来源**:`libs/common/` 下的 schema/模型,以及设计文档对应章节。核对漂移项时以此为准。
- **验证闸所在**:`eval/`(T1–T6)与契约 pin 测试(尤其 `chunk_id` pin)。这是判定"闭合"的唯一裁判。

若以上任一缺失或路径不明,先报告、不开工。

---

## 核心循环:逐行按"分类"三态处理

对账表的每一行,按其**分类**走下面的决策树。**先处理简单态(确认/跳过),把真正要写代码的"真缺"和"漂移"留到后面集中做**,这样账本进度清晰。

### 分类 = 完成

不写代码。可选地用 `codegraph_search` / 一次相关测试快速确认它确实在且接线正常,然后在账本上勾掉。**不优化、不重构。**

### 分类 = 触发驱动(最重要的护栏)

**绝不实现。** 这一项是"按设计有意未实施"的机制(如 Temporal、PostgresSaver、Nebula、全量解析器),它的正确状态就是一个带触发条件的 stub。实现它 = 把前面几轮刻意防住的过度工程招回来、破坏触发纪律。

只做核对:
- 用 `codegraph_search` 或 `grep` 找到该项的符号。
- **返回一个带 `NotImplementedError` + 触发条件 docstring 的 stub** → 正确,核对触发条件文字是否与设计文档/迁移记录一致,一致即勾掉。
- **stub 缺失**(既无实现也无标注)→ **只补 stub,不补实现**:建一个 `raise NotImplementedError`、docstring 写明触发条件的占位。这是"补上标注",不是"补上功能"。
- **返回的是真实实现**(有人把它实现了)→ 这是反向越界,停下报告,交人决定是否回退。

### 分类 = 真缺(在 M1/当前范围内)

写代码补上,**以测试驱动**(见下节验证闸)。补之前先 `codegraph_impact` 看波及半径,确认不会动到契约层或范围外的行。

### 分类 = 漂移

把代码**修正到符合契约值**,契约值本身一字不改。
- 若代码偏离了 `libs/common` / 设计文档里的契约 → 改代码,使其符合契约。
- 若**文档与代码对"契约值应该是什么"本身有分歧**(例如公式定义两处不一致)→ 这是权威之争,**停下,交人裁决**,不要替任何一方做主。

---

## 验证闸:测试不过 ≠ 闭合(validator → fix → repeat)

这是本 skill 的脊梁。**绝不以"我实现完了"作为闭合标准。一项差异只有在它对应的 eval / 契约 pin 测试通过时,才算闭合。**(注:编码 agent 没有可靠的"实现是否符合规格"自检能力,所以裁判必须是外部测试,不是自报。)

对每个"真缺"项执行:
1. 找它对应的测试。**没有就先写一个**——写完它应当**失败**(红),证明这项确实缺。
2. 实现该项。
3. 跑该测试。绿 → 闭合;红 → 修复,回到第 2 步重复。
4. 该项绿了之后,跑一次该项**契约 pin 测试**(若涉及契约层),确认没把契约值碰歪。

对"漂移"项:修正后必须让对应契约 pin 测试通过,且原本绿的不变红。

---

## 长清单的上下文卫生:派 forked subagent

差异列表一长,逐项实现会吃爆主上下文。**当待处理(真缺 + 漂移)项超过约 5 个时,逐项(或小批)委派给 forked subagent(Task 工具)处理。**

- 主会话**只维护账本**:对账表 + 每行状态。不在主会话里读大量源码。
- 每个 subagent 领一项(或一小批):读相关代码、写/跑测试、实现到绿、回报一段简短结果(改了什么文件、哪个测试、通过否)。
- 主会话收到回报后在账本勾掉该项,继续派下一项。

这样无论清单多长,主会话上下文始终干净。

---

## 收尾回归

全表处理完,**跑一次全量 `eval/` T1–T6 + 全部契约 pin 测试**。后面的修改可能打破前面的契约,只有全量回归能抓到。任一项变红 → 停下报告,不要带病收尾。

(可选)若项目里装有 `simplify` / code-reviewer 类只读评审 skill 且离线可用,对本次改动过的文件跑一遍质量评审。不可用则跳过。

---

## 收尾报告(ALWAYS 用此模板)

最后输出一份报告交人定夺,**严格用以下结构**:

```markdown
# 对账差异闭合报告

## 已补全(真缺 → 实现 → 测试绿)
- <文档项>:补了什么 | 把关测试 <测试名>(通过)

## 已修正(漂移 → 修正至契约)
- <文档项>:原本怎么偏的 | 改成符合 <契约来源> | 契约 pin 测试(通过)

## 已跳过 — 触发驱动(未实现,符合设计)
- <文档项>:触发条件 <…> | stub 状态(已存在/本次补了标注)

## 待裁决(未自行处理)
- <文档项>:具体含糊点(尤其契约权威之争:文档说 X、代码说 Y)

## 测试未过(已实现但未闭合)
- <文档项>:哪个测试红了 | 现象/疑因

## 范围外未动
- "完成"类共 N 项,均未改动
```

---

## 明确不做(Out of scope)

- 实现任何"触发驱动"项的真实功能(只补 stub 标注,功能等触发条件满足时另走 CP)。
- 改动任何契约值(chunk_id 公式、字段名、schema)。
- 业务模块开发、对范围外代码的重写或优化。
- 以"自报完成"代替测试通过来宣告闭合。
