# 第一期文档处理管线测试 —— 使用指南(团队版)

> 面向团队成员:**不需要懂管线内部**也能跑。本指南讲清楚 ① 怎么用 ② 怎么看报告 ③ 三大目标怎么下结论。
> 快速参考见 `README.md`;本文是详细版。

---

## 0. 这个脚本是干什么的

给一批 PDF(**已经按语料类型分好类、没有 manifest 登记表**),**跳过登记阶段**,逐件跑一遍文档处理管线的核心步骤
(`解析 → 结构化切块 → 质检 → 正则要素抽取`),用指标 + 可选大模型裁判,回答三个问题并出一份**易懂报告**:

1. **正则匹配覆盖度** —— 条款/要素能不能被正则成功结构化抽出来?哪些抽漏了?
2. **解析模块要不要上 DeepDoc** —— 现在的轻量解析(light)够不够?哪些文档版面破碎/扫描件需要 DeepDoc/OCR?
3. **QC 门控阈值要不要调** —— 7 个质检指标的实际数值分布,哪些阈值太严(误杀好文档)/太松(放过坏文档)?

报告会**标出管线失效的 PDF**,并给**异常指标分析**与**阈值调整建议**。

**什么时候用**:拿到新一批待入库语料、或想验证解析/正则/质检对新文档的适配度时,先跑这个评估,再决定要不要调管线。

## 1. 前置准备(一次性)

1. **Python 3.11 虚拟环境 + 依赖**(本仓约定,见根 `CLAUDE.md`):
   ```bash
   # 仓库根目录下,确保 .venv 是 python@3.11 建的,并已 editable 安装
   .venv/bin/pip install -e libs/common && .venv/bin/pip install -e pipeline
   ```
   验证:`.venv/bin/python -c "import pipeline, common; print('ok')"` 打印 `ok` 即可。

2. **大模型(LLM)**——目标②③ 的深度判断要用它(目标① 也用它核对正则对错)。走**环境变量**,**绝不写进配置/不入库**:
   ```bash
   export OPENAI_API_KEY="sk-..."                    # 必填(开 LLM 时)
   export OPENAI_BASE_URL="https://你的网关/v1"       # 可选,默认 https://api.openai.com/v1
   export OPENAI_MODEL="gpt-5.4-nano"                # 可选,也可在 config 里写 llm.model
   ```
   > 没有 key 也能跑——把 `config.yaml` 里 `llm.enabled` 设 `false`,只出确定性指标报告(目标②③ 会标注"需 LLM 复核")。

## 2. 第一步:准备 PDF 并分类

脚本**靠目录结构识别语料类型**(因为没有 manifest)。把 PDF 按类型放进子目录,例如:

```
<你的根目录>/
├── 内规/        *.pdf   → P-INT(内部制度)
├── 外规/        *.pdf   → P-EXT(外部法规)
├── 案例/        *.pdf   → P-CASE(监管处罚决定书/警示函)
└── 监管问答/    *.pdf   → P-QA(问答对语料)
```

四种 `corpus_type` 决定**测哪套正则、跑哪些质检指标**:

| 类型 | 代码 | 目标① 主测的正则 | 跑的 QC 指标 |
|---|---|---|---|
| 内规/外规 | `P-INT`/`P-EXT` | **条款树**(章/节/条/款/项)+ 文号/日期/机构 | 全部 7 项 |
| 案例 | `P-CASE` | **案例要素**(处罚机构/当事人/日期/金额/类型) | 页码锚点 + 文本质量(2 项) |
| 监管问答 | `P-QA` | 问答对检测 | 问答对完整率 + 页锚 + 文本质量(3 项) |

- 子目录名可以是任意中文/英文,只要在 `config.yaml` 的 `corpus_map` 里写好"目录名 → 类型"映射即可。
- 也支持 `file_overrides` 逐文件指定类型(优先级高于目录)。
- 没匹配到任何类型的 PDF:默认 **skip**(报告里会提示跳过了几件)。

---

## 3. 第二步:配置 `config.yaml`

```bash
cp tools/doc_test/config.example.yaml tools/doc_test/config.yaml
```
逐项说明(改你需要的):

```yaml
pdf_root: "/绝对路径/你的PDF根目录"   # 必填
recurse: true                          # 递归扫描子目录(false 只扫根目录第一层)

corpus_map:                            # 目录名 → 类型(按你的实际目录名改)
  内规: P-INT
  外规: P-EXT
  案例: P-CASE
  监管问答: P-QA
file_overrides: {}                     # 可选:{"某文件.pdf": P-CASE}
unclassified: skip                     # 未匹配的件:skip 跳过 | 或填一个类型兜底

parse:
  scanned_char_per_page_max: 50        # 字符/页 低于此 → 判扫描件(需 OCR)。一般不用改

qc_thresholds: {}                      # 留空=用仓库现有阈值;覆盖做实验,例:
                                       #   {clause_coverage_min: 0.90, text_garbled_max: 0.02}
llm:
  enabled: true                        # 是否开大模型裁判(目标②③ 深度)
  model: "gpt-5.4-nano"
  max_chars: 12000                     # 送模型的文本上限(控成本)
  sample: head_mid_tail                # 文本采样:head_mid_tail(首+中+尾) | head(只首部)
  max_pdfs: 0                          # >0 = 只对前 N 件跑 LLM(冒烟/控成本);0 = 全跑

output:
  dir: "tools/doc_test/out"            # 报告与 JSON 落盘目录
  report_name: "phase1_report.md"
  metrics_name: "phase1_metrics.json"
```

**控成本小贴士**:第一次先 `max_pdfs: 5` 跑 5 件看效果,确认无误再 `0` 全量;或先 `llm.enabled: false` 出确定性报告,再开 LLM。

## 4. 第三步:运行

```bash
.venv/bin/python tools/doc_test/run_phase1.py --config tools/doc_test/config.yaml
```

控制台会打印进度,最后给出落盘路径:
```
⚠ 未分类跳过 2 件(目录名不在 corpus_map):xxx.pdf …      # 如有
开始测试 40 件 PDF …
[1/40] P-EXT  某管理办法.pdf
...
完成。管线失效 3/40 · LLM 裁判 40 件
报告:tools/doc_test/out/phase1_report.md
指标:tools/doc_test/out/phase1_metrics.json
```
> 运行中可能刷 `Could not get FontBBox …` 之类的行——是 PDF 字体库的**无害警告**,不影响结果,忽略即可。

产物两份:
- **`phase1_report.md`** —— 人看的报告(下一节详解怎么看)。
- **`phase1_metrics.json`** —— 逐 PDF 的原始指标(机器可读,做阈值对比实验/二次分析用)。

## 5. 第四步:怎么看报告(重点)

报告分 6 块。下面逐块讲**怎么读数 + 看到异常怎么办**。

### 5.0 总览
```
- 总 PDF:40 · 按类型:P-EXT 25、P-INT 10、P-CASE 5
- 管线失效:3/40 (8%) · 解析失败:2/40 (5%)
- LLM 裁判:已启用
```
先看**管线失效占比**。失效 = 这份 PDF 没能正常走完管线(解析失败/切块为空/QC 未过)。占比高 → 这批语料整体适配差。

### 5.1 ⚠ 管线失效 PDF
逐件列出**失效文件 + 类型 + 原因**。常见原因:
- `解析失败 E202…疑似扫描件` → 这是**图片型/扫描 PDF**,light 抽不到文本 → 直接关联目标②(需 OCR/DeepDoc)。
- `切块产物为空(0 chunk)` → 解析出文本但条款树一条都没切出来 → 关联目标①(正则完全没接住该体例)。
- `QC 未过:…` → 走完了但某质检指标不达标 → 关联目标③。

**怎么办**:失效件是优先排查对象。先人工打开看一眼,确认是"文档本身的问题"(扫描件/格式怪)还是"管线的问题"(正则/阈值不适配)。

### 5.2 目标① 正则匹配覆盖度 / 结构化提取
三张信息:

**(a) 要素正则抽取命中率**(逐类型逐字段)
```
| 类型   | 字段        | 命中率     |
| P-EXT  | doc_number | 22/25 (88%) |
| P-CASE | amount_wan | 3/5 (60%)   |
```
**命中率 = 抽到该字段的件 / 总件**。低于预期(如文号只 88%)说明**正则在那几件上漏抽了**。
**怎么办**:去 `metrics.json` 找该字段为空的件名 → 打开 PDF 看它的文号/金额长什么样 → 多半是正则没覆盖的写法 → 反馈给正则维护者补规则。

**(b) 条款树覆盖**(仅内规/外规)
```
clause_coverage min 0.62 / 中位 1.00 / max 43.9 · 表头识别率 中位 0.26
```
- `clause_coverage` = **结构化条数 ÷ 宽松「第X条」命中数**。关键读法:
  - **>1 正常**:文档用小数/交易所编号体例(如 `1.1`、`2.3`),「第X条」分母很小,所以比值很大,不是问题。
  - **<1 才是问题**:正则**漏接了**该文档的条款编号体例(结构化条数少于实际)。报告**只列 <0.95 的件**在下面的小表里。
- `表头识别率` = 正文段落中被识别为 章/节/条/款/项 的比例。
**怎么办**:盯**覆盖偏低(<0.95)的件**——它们是正则没接住的编号体例,是补正则的高价值样本。

**(c) LLM 核对正则抽取**(开了 LLM 才有)
大模型读原文当"真值",标出**抽错的字段 / 漏识别的结构**,并给综合覆盖评分。
```
| 文件 | 抽错字段 | 结构完整 | 问题 |
| xx.pdf | date | 否 | 第十二条之一未被识别 |
```
**怎么办**:这是最直接的正则缺口清单——"问题"列就是要补的具体规则。

### 5.3 目标② 解析模块是否需要 DeepDoc
```
- 启发式:疑似扫描件/低文本密度 3/40 (8%) · 乱码率>1% 1/40
- LLM 判定 建议上 DeepDoc:6/40 (15%) · 原因分布:tables_lost×4、multi_column_interleaved×2
```
- **启发式**(不依赖 LLM):扫描件/低密度 = light 抽不到文本,**必须 OCR**;乱码率高 = 编码/字体问题。
- **LLM 判定**:读解析文本判断版面是否破碎,给"建议上 DeepDoc"的件 + 原因码:
  - `scanned_no_text` 扫描件无文本 · `tables_lost` 表格被压扁丢失 · `multi_column_interleaved` 多栏交错 · `ocr_garble` OCR 乱码 · `reading_order_broken` 阅读顺序乱 · `formula_lost` 公式丢失。
**怎么下结论**(报告末尾也有口径):
  - 扫描件/低密度占比高 → **必须接 OCR**(DeepDoc/PaddleOCR),light 解决不了;
  - 版面破碎/表格丢失(`tables_lost`/`multi_column…`)占比高 → **建议上 DeepDoc** 提升结构化质量;
  - 两类占比都低 → **light 解析够用**,先不上 DeepDoc。

### 5.4 目标③ QC 门控阈值是否需要调整
```
| QC 指标       | 当前阈值 | 数值分布                | 失败数 | 阈值建议            |
| 条款覆盖率     | 0.95    | min 0.6/中位 1.0/max 44 | 2     | ⬇ 疑似太严(3 好件被拦) |
| 文本质量       | 0.01    | min 0/中位 0/max 0.03   | 1     | ✓ 无失败            |
```
逐指标看:**实际数值分布 vs 当前阈值 vs 失败数 vs 建议**。
- **阈值建议**由 **LLM 判文档好坏 × QC 通过/失败** 的不一致推出:
  - `⬇ 疑似太严`:有 ≥2 份 **LLM 判"好"却被该指标拦下**的件 → 假阳性,阈值可能过严,考虑**放宽**。
  - `⬆ 疑似太松`:有 ≥2 份 **LLM 判"坏"却放过**的件 → 假阴性,阈值可能过松,考虑**收紧**。
- 7 个指标含义速查:

  | 指标 | 默认阈值 | 含义 |
  |---|---|---|
  | 条款覆盖率 clause_coverage | ≥0.95 | 结构化条数/宽松第X条(见 5.2 读法) |
  | 条号连续性 clause_continuity | ≤0(无缺口) | 条号是否连续(缺条数) |
  | 层级合法性 hierarchy_legality | ≤0 | 章/节/条 顺序非法处数 |
  | 页码锚点完整率 page_anchor_complete | ≥1.0 | 块带页码的比例(PDF 一般满足) |
  | 表格一致性 table_consistency | ≤0.05 | 空表占比 |
  | 文本质量 text_quality | ≤0.01 | 乱码字符占比 |
  | 抽取充分性 extraction_sufficiency | ≥0.7 | 各页字符密度均匀度(查漏页/解析不全) |
  | 问答对完整率 qa_pair_completeness | ≥0.95 | (仅 P-QA)完整问答对/问标记 |

**怎么办**:**建议项是线索不是结论**——按建议挑 2~3 个具体件人工复核,确认确属误杀/漏放,再决定改 `config/qc_thresholds.yaml`。可先用 `config.yaml` 的 `qc_thresholds:` 覆盖跑一版对比(见第 6 节),不必先改仓库。

### 5.5 异常指标分析
按 **QC 指标失败率**从高到低排行。失败率最高的指标 = 最值得关注的——要么这批文档在该维度普遍差,要么该阈值不合理。结合 5.4 的"阈值建议"一起判。

## 6. 阈值对比实验(目标③ 进阶)

想试"把某阈值放宽后失败分布怎么变",**不用改仓库配置**,在 `config.yaml` 覆盖即可:
```yaml
qc_thresholds: {clause_coverage_min: 0.90, extraction_sufficiency_min: 0.6}
```
改完重跑,对比两版报告的"失败数/阈值建议"。`metrics.json` 里每件的指标**原始数值**也都在,可自己拉去画分布。

## 7. 常见问题(FAQ)

- **"没有可测 PDF"** → 检查 `pdf_root` 路径、`corpus_map` 目录名是否和实际一致、`recurse` 是否需要开。
- **"未分类跳过 N 件"** → 这些 PDF 的父目录名不在 `corpus_map` 里。补映射,或用 `file_overrides`,或把 `unclassified` 设成一个兜底类型。
- **"⚠ LLM 不可用,降级为仅确定性指标"** → `OPENAI_API_KEY` 没设或网关不通。目标②③ 会少 LLM 部分;设好 key 重跑即可。
- **刷 `FontBBox` 警告** → 无害,PDF 字体元数据问题,忽略。
- **想省钱** → `llm.max_pdfs: 5` 先冒烟,或 `llm.enabled: false` 先看确定性报告。
- **报告/配置会不会进 git** → 不会,`tools/doc_test/out/` 和 `tools/doc_test/config.yaml` 已 gitignore;`config.example.yaml` 是模板,入库。

## 8. 三大目标:报告 → 结论 速查

| 目标 | 看报告哪里 | 怎么下结论 |
|---|---|---|
| **① 正则覆盖度** | 5.2 命中率表 + 覆盖偏低件 + LLM 核对"问题"列 | 命中率低的字段 / clause_coverage<0.95 的件 / LLM 标的漏识别 = **要补的正则规则清单** |
| **② 要不要 DeepDoc** | 5.3 扫描件占比 + LLM 建议占比与原因 | 扫描件多 → 必接 OCR;版面破碎/表格丢失多 → 建议 DeepDoc;都低 → light 够用 |
| **③ QC 阈值** | 5.4 数值分布/失败数/建议 + 5.5 失败率排行 | 按"⬇太严/⬆太松"建议挑件人工复核,再决定改 `config/qc_thresholds.yaml`;可先 config 覆盖跑对比 |

有疑问或想加指标(如表格还原度、特定字段的精确率)直接提,脚本是模块化的(`metrics.py` 确定性 / `judge.py` 大模型 / `report.py` 报告),好扩展。
