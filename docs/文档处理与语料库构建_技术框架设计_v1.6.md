# 文档处理与语料库构建：技术框架设计方案（V1.6）

> 适用项目：东方证券审计大模型系统一期
> 覆盖范围：原始文件接入 → 解析 → 质检 → 结构化切块 → 元数据/版本链 → 向量化 → 索引入库 → 语料治理（即《制度查询与制度比对智能体_RAG技术框架设计》§3 摄取管线的专项展开）
> 语料规模：内规 ≈ 1,000 件，外规（含监管问答）≈ 20,000 件，监管案例 ≈ 10,000 件，估算 60–95 万 chunk
> 设计依据增补：项目启动会修订版（2026-06）+ 甲方提问方式与应用场景样本（2026-06，驱动 §19.1 期限单位归一化、§19.2 实体类型打标维度）
> 文档性质声明：【标准】= 行业成熟实践；⚠ = 合成启发式/工程估算，须实测标定，对甲方陈述前经张翼飞复核
> V1.6 增量：CP-007 联动（E1 期限单位归一化 / E2 实体类型打标维度 / entity_type schema）+ CP-008（§22 一期数据摄取全景的摄取侧承接：P-MISC profile + 三类非文档摄取的边界声明）

## 0. 设计目标与硬约束

| # | 目标/约束 | 来源 | 在本设计中的体现 |
|---|---|---|---|
| 1 | 未过质检的文档**不得**进入检索库 | 启动会"文件解析风险"应对 | §5 质检硬关卡 + 人工补录队列 |
| 2 | 每个 chunk 必须可四级回溯（条款→文档→页码→版本） | 依据验收 | §6 页码锚点 + §10 PG 权威条款表 |
| 3 | 数据流单向只读，知识库更新仅限人工导入 | 项目红线 | 全管线无任何源系统回写；导入入口唯一（§3） |
| 4 | 权限、密级、操作日志全程留痕 | 权限验收/安全验收 | §3 登记即定级 + §14 审计日志 |
| 5 | 支持 Word/PDF/Excel/图片/扫描件 | 一期支撑工作 | §4 格式分支 |
| 6 | 长任务异步 + 状态可见 | 性能验收 | §11 Temporal 编排 + 状态机 |
| 7 | PG 为权威存储，Milvus 可随时全量重建 | 工程原则 ⚠ | §12 一致性与重建 |
| 8 | Schema add-only，复用既有工厂/抽象 | 既定 CC 约束 | §10 表设计向后兼容 |

**一句话架构**：以 PostgreSQL 为权威事实库、Milvus 为可重建的检索投影、Temporal 为带人工关卡的编排骨架，管线对三类语料执行**同一骨架、不同处理档案（Processing Profile）**。

## 1. 总体管线架构

```
                        ┌──────────────────────────────────────────────┐
                        │ 治理层：批次管理/对账/质量看板/golden set 回归    │
                        └──────────────────────────────────────────────┘
 ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐
 │S0 接入  │→ │S1 解析  │→ │S2 质检  │→ │S3 结构化│→ │S4 元数据│→ │S5 索引  │→ [INDEXED]
 │登记/去重│   │DeepDoc │   │硬关卡   │   │切块     │   │版本链   │   │向量+PG │
 │定级/批次│   │OCR分支 │   │↘人工队列│   │条款树   │   │↘人工确认│   │双写     │
 └────────┘   └────────┘   └────────┘   └────────┘   └────────┘   └───┬────┘
                                                                      │(仅比对范围语料)
                                                                  ┌───▼────┐
                                                                  │S6 图谱  │
                                                                  │抽取队列 │→ 专家审核 → 图入库
                                                                  └────────┘
```

### 1.1 文档级状态机（PG `documents.pipeline_status`）

```
REGISTERED → PARSING → QC_PENDING ─┬→ STRUCTURING → META_REVIEW ─┬→ EMBEDDING → INDEXING → INDEXED
     │           │                 │                              │
     │           ▼                 ▼                              ▼
     │      PARSE_FAILED      QC_FAILED ──→ REMEDIATION ──→ (重回 QC_PENDING)
     │                                       (人工补录)
     ▼
 QUARANTINED (重复/命名不合规/密级待定/疑似病毒文件)

V1.1 增补：INDEXED 不是单向终态——
 INDEXED ──(质量工单确认, §18)──→ REPARSE_PENDING → REPARSING(全管线重跑) → INDEXED
                                        │ 重跑期间旧 chunk 保持在线但标 quality_flagged
                                        └→ 重跑仍失败 → 降级或退回甲方(同 §5.2 处置)

终态附加位（与版本生命周期正交）：status ∈ {effective, superseded, abolished, upcoming}
```

状态机每次迁移写 `pipeline_events` 表（时间、操作者/系统、前后状态、错误码），即数据侧操作日志。

### 1.2 设计上的三个关键取舍

1. **质检是阻断点而非告警点**：QC_FAILED 文档停在人工队列，绝不"带病入库"。代价是入库进度受人工补录吞吐制约——通过分层抽检率控制人工量（§5.4），而不是放松关卡。
2. **PG 先行，Milvus 后置**：S3 结构化完成即写 PG 条款表（权威），S5 才写向量。任何时刻 Milvus 可由 PG + embedding 服务全量重建，向量库不承担数据安全责任。
3. **人工环节用 Temporal Signal 建模**，不用轮询脚本：补录完成、元数据确认、图谱审核都是向挂起的 workflow 发 signal，工作流可挂起数天且断点精确。

## 2. 三类语料的处理档案（Processing Profile）

同一管线骨架，按 `corpus_type` 加载不同档案。档案 = 解析配置 + 切块规则 + 元数据 schema + 质检阈值 + 抽检率的组合。

| 配置项 | P-INT 内规 (1k) | P-EXT 外规 (20k) | P-QA 监管问答 (外规子类) | P-CASE 案例 (10k) |
|---|---|---|---|---|
| 解析主通道 | DeepDoc 全要素 | DeepDoc 全要素 | DeepDoc + QA 对分割器 | DeepDoc + 要素分段器 |
| 切块单元 | 条款级（§6.2） | 条款级 | **问答对**（一问一答=1 chunk） | 要素段（事实/依据/决定）+ 全文摘要块 |
| 父子块 | 是（父=节） | 是 | 否（QA 自完备） | 否 |
| 版本链 | 完整（修订说明人工录入） | 完整（新旧版 diff 供比对 Pass 1） | 按发布批次，无逐条版本 | 无版本，按处罚日期 |
| 元数据人工确认 | **逐件确认**（密级/业务域/状态） | 抽样确认 + 规则批量 | 规则批量 | 规则批量 + 字段完整率统计 |
| 质检方式 | 100% 规则校验 + 10% 人工抽检 ⚠ | 100% 规则校验 + 分层 3% 人工抽检 ⚠ | 规则校验（QA 对完整性） | 字段完整率 ≥ 阈值 + 2% 抽检 ⚠ |
| 图谱纳入 | 是（全量，懒构建触发） | 仅制度映射表命中子集 | 否 | 否（结构化字段替代） |
| 入库优先级 | W1（最先，建立基准） | W2（分层灰度） | W2 末段 | W3 |

> 外规 20k 内部再按 `sub_type` 分层：法律 / 行政法规 / 部门规章 / 规范性文件 / 自律规则 / 监管问答。层级决定 `issuer_level` 字段取值与比对智能体的义务来源优先级，也决定灰度放量顺序（§13.2）。

> **V1.6 增补（CP-008）**：监督共享信息（功能 5）的部门输出文档（合规事项 / 问责情况 / 监管函件 / 法律诉讼 / 风险事项）无条款结构，需第五个 profile **P-MISC**——同一管线骨架，但禁用条款树构建、改段落级切块、元数据 schema 重写。P-MISC 完整定义见 §22.2。费用结构化数据（功能 6/7）、审计项目资料（功能 8）、模板（功能 9）三类不走文档切块管线，边界声明见 §22.3–§22.5。

## 3. S0 文件接入与登记

### 3.1 导入入口（唯一）

一期全部走**文件导入**（接口预留不启用）。入口为知识库管理页面的批量上传，每批次必须随附**导入清单（manifest）Excel** ⚠：

| Manifest 必填列 | 说明 | 校验规则 |
|---|---|---|
| 文件名 | 与实际文件一一对应 | 不匹配 → 整批拒收 |
| 语料类型 | internal / external / qa / case | 枚举校验 |
| 子类型 | 见 §2 sub_type 枚举 | 枚举校验 |
| 标题 / 发文字号 | 案例填处罚决定书文号 | 发文字号格式正则（如"沪国资委审计〔2025〕114号"）⚠ |
| 发布机构 | — | 机构字典表匹配，未命中 → 待确认 |
| 发布日期 / 生效日期 | yyyy-mm-dd | 日期合法性；生效 ≥ 发布告警 |
| 版本说明 | 新文件 / 替代某文号 / 废止某文号 | 替代/废止须填被替代文号，§8 版本链用 |
| 密级 | 公开 / 内部 / 机密（按甲方分级口径，待确认） | 缺失 → QUARANTINED |
| 业务域 | 多值，业务域字典表 | 未命中 → 待确认 |

文件命名规范 ⚠：`{语料类型}_{发文机构简称}_{发文字号或短标题}_{版本日期}.{ext}`。不合规不阻断（manifest 为准），但记入批次质量报告。

### 3.2 登记动作（事务内完成）

1. **计算 SHA-256 file_hash**：
   - 精确重复（hash 命中已有记录）→ 跳过并在批次报告标注"重复，关联已有 doc_id"
   - 疑似重复（标题 + 发文字号命中但 hash 不同）→ QUARANTINED，人工裁决"新版本 / 同文不同渠道 / 误传"
2. 生成 ID：`doc_logical_id`（ULID，制度逻辑实体，跨版本不变）+ `doc_version_id`（ULID，本次文件）。manifest 声明"替代某文号"时，logical_id 继承被替代文档的
3. 原始文件落 MinIO：`raw/{corpus_type}/{import_batch_id}/{doc_version_id}.{ext}`，只写一次永不修改（原件留证）
4. 写 `documents` + `import_batches` 表，pipeline_status = REGISTERED
5. 格式探测（magic number，不信任扩展名）：doc/docx/pdf(文本层)/pdf(无文本层=扫描)/xlsx/jpg/png；超出白名单 → QUARANTINED

### 3.3 批次模型

`import_batches`：batch_id、操作人、时间、manifest 文件引用、文件数、各状态计数、批次质量报告引用。**批次是回滚单元**（§12.3）。

## 4. S1 解析层

### 4.1 路由

```
docx/doc ──────────────→ DeepDoc(office通道)
pdf 有文本层 ───────────→ DeepDoc(pdf通道)
pdf 无文本层 / 图片 ─────→ PaddleOCR(GPU) → 版面重建 → DeepDoc 后处理
xlsx ──────────────────→ 表格直读(openpyxl) → 表格IR
复杂版式解析失败兜底 ─────→ MinerU 重试一次 → 仍失败 → PARSE_FAILED
```

判定"有无文本层"：抽取字符数 / 页数 < 50 字符/页 ⚠ 视为扫描件走 OCR。

### 4.2 解析输出：统一中间表示（IR）

所有通道归一为同一 IR JSON（存 MinIO `ir/{doc_version_id}.json`，供 S2/S3 消费与问题回放）：

```json
{
  "doc_version_id": "...",
  "page_count": 18,
  "blocks": [
    {"block_id": "b0012", "type": "heading|para|table|figure",
     "level": 2, "text": "第三章 费用报销管理",
     "page": 5, "bbox": [x0,y0,x1,y1], "ocr_conf": 0.97}
  ],
  "tables": [
    {"table_id": "t003", "page": 7, "rows": 12, "cols": 5,
     "cells_md": "|费用项目|标准|...|", "anchor_block": "b0031"}
  ]
}
```

要点：
- **bbox + page 必须保留**——扫描件的页码锚点来自 OCR 坐标，这是四级引用在扫描件上成立的唯一途径
- `ocr_conf` 块级置信度参与质检（§5.1 指标 6）
- 表格统一序列化为 markdown 单元格矩阵，合并单元格展开补值（DeepDoc 能力），原始结构保留在 cells 元信息中

### 4.3 解析并发与资源隔离

- DeepDoc/CPU 解析池与 OCR/GPU 池为两条独立 Temporal task queue，互不抢占
- 单文档解析超时 ⚠：常规 5 min，扫描件 15 min；超时 → PARSE_FAILED(E203)
- 解析 worker 无状态、可水平扩，IR 落盘后 worker 即可释放

## 5. S2 解析质检（硬关卡）

### 5.1 规则校验指标（100% 执行，机器自动）

| # | 指标 | 计算方法 | 阈值 ⚠（V0 标定，下列为初值） |
|---|---|---|---|
| 1 | 条款覆盖率 | 结构化识别出的"第X条"数 ÷ 全文正则扫描到的"第X条"标记数 | ≥ 95% |
| 2 | 条款序号连续性 | 条号序列缺口数（第7条后直接第9条） | 缺口 = 0（允许"第X条 删除"显式占位） |
| 3 | 标题层级合法性 | 章→节→条 层级倒挂块数 | = 0 |
| 4 | 页码锚点完整率 | 有 page 值的 block 占比 | = 100% |
| 5 | 表格还原合理性 | 各表 行×列 与单元格数一致性；空表占比 | 空表 ≤ 5% |
| 6 | 文本质量 | 非 CJK 乱码字符占比；OCR 文档块均值 ocr_conf | 乱码 ≤ 1%；ocr_conf ≥ 0.85 |
| 7 | 抽取充分性 | 抽取字符数 ÷ (页数 × 中位密度) | ≥ 0.7 |

任一不达标 → QC_FAILED，写明**失败指标 + 定位证据**（如"第9-11条缺失，疑似第8页表格吞并正文"），进入补录队列。

### 5.2 人工补录队列

补录工作台（知识库管理页的子模块）能力：
- IR 与原文 PDF 并排视图，失败指标高亮定位到页
- 三种处置：**修正后重跑**（人工修正 IR 块/补条款边界 → 重入 QC）/ **降级入库**（标记 `degraded=true`，仅全文检索可见、不参与条款级引用——适用于无法结构化的历史扫描件）⚠ / **退回甲方**（要求提供可编辑版原件，对应启动会"要求甲方提供"类应对）
- 每次处置写 `remediation_records`（操作人、动作、耗时），为后续优化解析器提供失败样本库

### 5.3 人工抽检（在规则校验通过的文档上叠加）

按 §2 档案抽检率分层抽样，抽检项：条款边界正确性、页码锚点准确性（随机抽 3 个 chunk 与原文对页核对）、表格语义还原。抽检不合格率 > 5% ⚠ → 该批次同 sub_type 文档全部回退重检，并触发解析配置调整。

### 5.4 Golden Set 回归

W1 期间从内规中选 50 件结构代表性文档 ⚠，人工标注完整条款树作为金标准。此后**任何解析器配置/版本变更**必须先跑 golden set 回归（条款树 F1 ≥ 0.98 ⚠）才允许上生产——防止"修好一类版式、打坏另一类"。

## 6. S3 结构化与切块

### 6.1 条款树构建

从 IR heading/para 块流构建条款树，节点类型与识别正则：

| 节点 | 正则（锚定行首） | 备注 |
|---|---|---|
| 章 | `^第[零一二三四五六七八九十百]+章` | |
| 节 | `^第[零一二三四五六七八九十百]+节` | 可缺省 |
| 条 | `^第[零一二三四五六七八九十百千]+条` | **原子引用单元** |
| 款 | 条内自然段（无显式编号，按段序计数） | 第N款 = 第N个自然段 |
| 项 | `^（[一二三四五六七八九十]+）` 或半角括号变体 | |
| 目 | `^[0-9]+[.、]` | |
| 附则/附件 | `^附\s*[则件]` | 独立子树；附件表格挂此节点 |

中文数字 → 阿拉伯数字归一化生成 `clause_path_norm`（如 `3.21.2` = 第三章第二十一条第二款），供排序、diff 对齐和案例条款反查的精确匹配。

边界情况处理 ⚠：
- 无章节直接列条的短文（常见于通知类）：虚拟根节点，条直接挂根
- "第X条之一"插入条：path 记 `21bis`，norm 记 `21.1b`，保持排序稳定
- 条文中引用其他条款（"依照第十五条"）：S3 阶段用正则捕获为 `internal_refs[]` 存条款表——这是图谱抽取（S6）和比对交叉验证的廉价前置信号

### 6.2 切块规则（P-INT / P-EXT）

| 规则 | 内容 |
|---|---|
| 原子单元 | **条**。一条一块为默认 |
| 超长条（> 600 token） | 按款/项拆分为多块，每块前缀重复条头（"第二十一条 ……（续）"），`seq` 递增 |
| 超短条（< 50 token） | 不合并（合并破坏引用精度），保持独立块 |
| 父块 | 节级（无节则章级），上限 2,000 token，仅入 PG 不入向量库 ⚠——检索命中子块后回查父块供 LLM 完整上下文（"小块检索、大块供证"） |
| 表格 | 独立 chunk（`chunk_type=table`），内容 = 表格 markdown + 一句 LLM 生成的表格摘要前缀 ⚠；挂接最近条款节点为 parent；> 30 行的大表按行组拆分，每块重复表头 |
| 上下文前缀 | 每块向量化文本 = `《文件标题》(发文字号, 版本日期) > 第X章 章名 > 第X条` 面包屑 + 条文正文。【标准】上下文化切块（Anthropic Contextual Retrieval 思路的轻量版：用结构面包屑替代 LLM 生成上下文，零额外调用） |
| 页码锚点 | `page_start/page_end` 取该条全部 block 的页跨度；扫描件取 OCR bbox 所在页 |

目标块长 300–600 token，与 bge-m3 编码效率区间匹配。

### 6.3 切块规则（P-QA 监管问答）

一问一答 = 1 chunk，问题文本加权进面包屑（问答检索以问句匹配为主）。QA 边界识别：`^问[:：]` / `^答[:：]` / 编号问答体 ⚠；边界识别失败计入质检指标 1 的等价项（QA 对完整率 ≥ 95%）。

### 6.4 切块规则（P-CASE 案例）

处罚决定书按要素分段：当事人段 / 违规事实段 / 处罚依据段 / 处罚决定段（段首模式 + LLM 辅助分段 ⚠）。每段一块 + 一个全文摘要块（LLM 生成 ≤ 150 字，`chunk_type=case_summary`，相似案例检索的主命中面）。要素结构化抽取见 §9。

### 6.5 chunk_id 设计（幂等的根基）

```
chunk_id = sha1(doc_version_id + "|" + clause_path_norm + "|" + seq)[:24]
```

确定性 ID 保证：同一版本重跑管线产生相同 ID → Milvus upsert 天然幂等、PG ON CONFLICT 可重入，任何阶段失败重试不产生脏数据。

### 6.6 图谱抽取的输入粒度解耦（CP-001，已批准）

> 问题：超短条独立成块（§6.2）服务引用精度，但条款级 chunk 直喂图谱抽取会导致短条实体召回差、指代悬空、跨条关系丢失。解法：**抽取粒度与检索/引用粒度解耦**——chunk 是引用单元，不是抽取单元。

1. **抽取窗口 = 节级父块**（≤2,000 token，无节则章级；超限按条滑窗、**重叠 1 条** ⚠ 防边界关系丢失）。"DeepDoc 切块替换 LightRAG 内置算法"的口径修正为：DeepDoc 交付 chunk **及其结构树**，抽取窗口由结构树装配，不逐 chunk 直喂。
2. **确定性指代预解析**：窗口渲染时按 §6.7 的 clause_references 标注内联注释（"前款"→〖即第三十一条第一款〗、"本办法"→〖文档标题+文号〗），窗口头部注入文档级元数据，短条实体天然带归属。零 LLM 成本。
3. **三元组条款级锚定与校验**：抽取 prompt 强制每条三元组带来源 clause_path；系统校验该 path ∈ 本窗口条款集，失败者降权进专家审核队列（不静默入图、不直接丢弃）。图谱 POC 的"引用准确率 ≥95%"自此明确为**三元组条款锚点校验通过率**——引用追溯链不因窗口放大而降级到节级。
4. **样板条款跳过清单** ⚠：解释权归属 / 施行日期 / 废止声明类条款（正则：`本(办法|规定|细则|指引).{0,6}(负责)?解释`、`自.{0,12}(施行|实施|执行)`、`同时废止`）不进抽取队列——其信息已由元数据与版本链承载（且后者经人工确认，比 LLM 抽取可靠），进图只产生噪声并侵蚀"关系校准 <3min"指标。跳过动作写日志，清单可维护。
5. 既有两道闸门保留为最终过滤器：本体硬校验（5 节点 + 6 关系枚举外机器直接拒绝）、专家三态审核。

整体为漏斗结构：#4 砍输入 → #1/#2 使输入可抽 → #3 使输出可校验 → 本体拒出格 → 专家面对短而干净的队列，最贵的专家人时单位产出最大化。
否决的备选：仅为图谱合并短 chunk（破坏引用单元一致性且解不了跨块指代）；全文级抽取（窗口过大召回衰减、锚点校验鉴别力退化 ⚠）。

### 6.7 指代预解析实现规格：ref_resolver（CP-001 实现细化，已批准）

S3 条款树建成后运行的**纯规则查表库**，零 LLM 调用。核心实现决策：**standoff 标注**——`chunks.text` 保持逐字原文，解析结果存独立表，带注释文本仅在 S6 窗口装配时临时渲染、不落库。

**四类指代模式（最长匹配优先，复合指代如"本办法第十五条第二款"整体匹配）：**

| 类型 | 模式 | 解析方法 |
|---|---|---|
| R1 文档自指 | `本(办法\|规定\|细则\|指引\|制度\|通知)`、本条/本章 | documents 元数据 / 当前树节点 |
| R2 相对条款 | 前条/前款/前两款/上述各款 | 条款树位置运算：前款=同条上一款（首款命中→UNRESOLVED 并计数，疑似原文笔误回灌 §18 工单）；**前条按文档序取、允许跨节**（法规行文惯例） |
| R3 绝对条款 | `第[中文数字]+条(第[中文数字]+款)?`、区间形 | 复用 §6.1 归一化器查 clause_path_norm；**作用域规则** ⚠：附件自带编号体系时优先在最近编号作用域解析，未命中上浮正文树 |
| R4 跨文档 | `《…》(（…〔YYYY〕N号…）)?(第X条)?` | 三级匹配：文号精确 → 标题精确 → **简称别名表 dict_aliases**（人工维护）；均未命中 → `pending_target`，夜间任务全量重试（随 W2 外规分批入库逐步收敛）；永久未解析项导出即**语料缺口清单**（供金总补采） |

**新表 `clause_references`**（add-only；原 `chunks.internal_refs[]` 保留不删、停止新写）：ref_id、chunk_id、doc_version_id、span_start/end、surface_text、ref_type(R1–R4)、target_doc_version_id、target_clause_path_norm、resolution_status(resolved/unresolved/ambiguous/pending_target)、method(现阶段恒为 rule，字段预留以禁止未来混入不可区分的 LLM 解析结果)。

**窗口渲染规则**：按 span **倒序**插入注释（防偏移漂移）；gloss ≤30 字；UNRESOLVED/ambiguous **不渲染**（宁缺勿错）；窗口 token 预算按 +15% 注释膨胀预留 ⚠。

**复用面（三处）**：S6 窗口渲染；比对智能体交叉验证（内规声称"依据《X》第N条"直接查表验证，免检索）；查询智能体引用展示的条款跳转（二期）。

**质量保证**：模式词典分类单元测试；golden set 50 件扩展指代解析标注（每件 5–10 处 ⚠）纳入回归；看板新增解析覆盖率与抽样准确率，R2 首款异常单列。

### 7.2 版本链登记

`doc_versions` 关系类型（add-only 枚举）：

| 关系 | 语义 | 触发后动作 |
|---|---|---|
| revise_replace | 修订替代（最常见） | 旧版 status→superseded；修订说明录入 |
| abolish_only | 废止不替代 | 旧版 status→abolished |
| merge_replace | 多文合并为一文 | 多个旧 logical_id 挂同一新版本（多对一） |

### 7.3 新版本到达的完整流转（增量更新主路径）

```
manifest 声明"替代〔2023〕45号"
 → S0: 继承 logical_id, 新建 version_id, 全管线正常跑
 → INDEXED 后执行原子切换事务:
    ① PG: 新版本 status=effective; 旧版本 status=superseded
    ② Milvus: 旧版本全部 chunk 的 status 标量批量更新为 superseded（不删除）
    ③ doc_versions 写入关系记录
 → 触发下游通知: 比对智能体的 Pass 1 候选队列（外规新旧版diff）;
   审计规范生成的制度变更打标队列（跨场景复用点）
```

旧版本向量**保留不删**：默认检索过滤 `status=effective` 已将其屏蔽，而变更查询、历史版本问答需要它在库。存储代价可忽略（月度更新量级数十件）。

## 8. S5 向量化与索引写入

### 8.1 Embedding 服务

- 调用甲方网关部署的 bge 系最新向量化服务（dense+sparse 一次产出，CP-005），离线摄取与在线查询**钉死同一模型版本与 endpoint**（训练-服务一致性【标准】）
- **新增运维依赖（CP-005）**：甲方升级 embedding 模型版本将使全库向量与 PG 冷备同时失效，须全量重新编码重灌——要求网关侧建立版本变更提前通知机制，重灌预案写入部署运维文档；sparse 双输出兼容性为接入前置验证项（CP-005-②衍生）

## 12. 一致性、重建与回滚

### 12.1 一致性模型

PG = 权威；Milvus = 投影。写顺序固定：PG chunks → Milvus upsert → flush → PG 状态置 INDEXED。中途崩溃的最坏结果是"PG 有、Milvus 无 + 状态非 INDEXED"= 对检索不可见的半成品，重试即收敛，**不存在 Milvus 有而 PG 无的脏向量**（chunk_id 确定性保证重跑覆盖）。

### 12.2 夜间对账任务

逐 doc_version 比对 PG chunk 数 vs Milvus `query(count)` 结果；不平 → E701 告警 + 自动修复（以 PG 为准重灌该文档向量）。对账报告进质量看板。

### 12.3 回滚

- **批次回滚**：按 import_batch_id 软删 PG 记录 + 按 doc_version_id 列表删除 Milvus 向量（PK delete）+ MinIO 原件保留（留证不删）
- **全量重建**：`rebuild` 命令从 PG chunks 重灌整个 collection，95 万 chunk 预计 2–3 小时 ⚠（embedding 可跳过——dense/sparse 向量同时落 PG bytea 冷备 ⚠，重建只是重写 Milvus，无需重新编码）

### 12.4 备份口径

PG 每日全备 + WAL；MinIO 原件异机副本；Milvus **不备份**（可重建）。该口径写入部署运维文档（启动会交付物）。

## 13. 容量、吞吐与全量摄取排期 ⚠（全节为工程估算，V0 后修正）

### 13.1 计算量测算

| 项 | 估算 |
|---|---|
| 总页数 | 1k×15 + 20k×10 + 10k×3 ≈ **24.5 万页** |
| DeepDoc 解析 | 12 worker × ~1 页/s → 纯计算 ≈ 6 h |
| OCR（扫描件占比未知，按 15% 假设）| 3.7 万页 ÷ (4 GPU worker × 1.5 页/s) ≈ 1.7 h |
| Embedding | 95 万 chunk ÷ 250 chunk/s ≈ 1 h |
| Milvus 写入 + 建索引 | < 2 h |
| **机器时间合计** | **≈ 1.5 个工作日（含重试与排队）** |

**真正的排期主导是人工环节**：内规逐件元数据确认（1k 件 × 2–3 min ≈ 5 人日）、10% 人工抽检（100 件 × 10 min ≈ 2 人日）、补录队列（失败率按 5% 估 ≈ 1,500 件外规级处理）。

### 13.2 分阶段入库计划（日历 3 周）

## 18. 解析逃逸处理闭环（V1.1 增补）

> 问题定义：**解析逃逸** = 文档通过 §5.1 全部规则校验进入 INDEXED，但实际解析质量差。V1 的抽检只在批级兜底（不合格率 >5% 回退整批），件级存在盲区，且入库后无修复回路。本节补齐。

### 18.1 已识别的四类盲区（规则指标为何看不见）

| # | 逃逸类型 | 为何穿透 7 项指标 | 审计场景危害 |
|---|---|---|---|
| A | 阅读顺序错乱（双栏串列、文本框/脚注插入正文） | 文字全在、字符数正常、条号连续；指标 1 的分子分母同源于已抽取文本，是自洽盲指标 | 条文语义被拼接污染，检索命中后引用展示错乱 |
| B | OCR 语义级错字（数字、否定词） | 块级 ocr_conf ≥ 0.85 仍容纳字符级高危替换（"5 日"→"50 日"、"不得"→"不许"） | **最高危**：直接改变义务含义，污染比对判定 |
| C | 表格语义畸变（合并单元格错位、表头错行） | 指标 5 只校验 行×列 一致性与空表率 | 费用标准、限额类表格数值张冠李戴 |
| D | 页眉页脚/水印混入条文 | 结构指标无感 | 条文夹杂噪声，embedding 质量下降 |

### 18.2 入库前增强检测（压缩逃逸率，承认不可归零）

1. **边缘通过带（marginal pass band）** ⚠：任一指标落在 [阈值, 阈值+ε] 区间（如条款覆盖率 95–97%、ocr_conf 0.85–0.90）即判"边缘通过"，**自动升级为人工抽检对象**（不计入常规抽检配额）。逃逸件在边缘带的密度远高于均匀分布，这是最便宜的捕获面。
2. **页眉泄漏与句完整性检测**（规则，全量，纳入质检指标 8/9 ⚠）：
   - 跨页重复短串检测：在 ≥60% 页面重复出现的行首/行尾短串视为页眉页脚，若其出现在条文 chunk 内部 → 命中类型 D
   - 句边界完整性：chunk 文本以非终止符截断且无续块衔接的比例 > 2% → 疑似类型 A
3. **双解析器仲裁** ⚠（定向，非全量）：对扫描件全量 + 边缘通过带文档，用 MinerU 做第二路解析，对齐两路条款树与正文（条款级文本相似度 < 0.92 的条目数 > 5% → 转人工）。成本可控：仲裁范围估算 < 全量的 20%。
4. **高危 token 复核**（仅 OCR 文档，规则 + 定向二次识别）：对条文中的数字串、金额、期限、否定词（"不得/禁止/不予"）做词表定位，该区域 OCR 置信度单独校验（≥ 0.95 ⚠），不达标触发该局部的二次识别或人工核对。类型 B 的针对性防线。

### 18.3 运行期反馈回路（捕获注定逃逸的残余）

入库前手段无法归零，剩余逃逸将在**使用时**暴露，工作流必须有接收通道：

| 信号源 | 机制 | 产物 |
|---|---|---|
| 用户纠错 | 查询/比对结果页每条引用旁设"引用有误"按钮，一键带 chunk_id 上报 | 质量工单（quality_tickets 表） |
| 生成侧自检 | 主答模型的引用约束 prompt 增加一项输出位：上下文存在乱序/截断/乱码时置 context_quality_flag ⚠ | 同一 chunk 被标记 ≥3 次 → 自动开工单 |
| 检索侧统计 | 周期任务：高频被召回但重排分数持续垫底、或从未被引用的 chunk 聚类分析 ⚠ | 低优先稽核清单 |
| 事后稽核 | 每周按**使用频次加权**抽样 INDEXED 文档复检（高流量文档优先享受复检资源） | 稽核报告 |

`quality_tickets`（add-only 新表）：ticket_id、chunk_id、doc_version_id、信号源、现象描述、状态（open / confirmed / reparsing / closed / rejected）、处置人、关联 REPARSE 任务。

### 18.4 REPARSE 工作流与下游失效传播

工单确认后触发 `DocReparseWorkflow`：

```
确认工单 → 文档转 REPARSE_PENDING
 → 旧 chunk 全部标 quality_flagged=true（仍在线可检索，引用展示处附"该来源正在质量复核"标识 ⚠）
 → 全管线重跑（人工可先修正 IR，同补录工作台）→ 重新过 QC（含 §18.2 增强项）
 → 原子替换（按 doc_version_id 范围）:
    ① PG: 旧 chunk 软删（保留 parse_revision 谱系供审计追溯），新 chunk 写入，parse_revision+1
    ② Milvus: delete by doc_version_id → upsert 新 chunk → flush
    ③ 文档回到 INDEXED，工单 closed
 → 下游失效传播:
    a. comparison_findings 中依据锚点落在受影响 chunk 的条目 → 标 stale=true，
       比对结果页显示"依据来源已修正，本条结论待复核"（不静默改结论——结论留痕原则）
    b. 已导出的 Excel 不追溯（导出即时间点快照），但导出记录关联的 chunk 谱系可查
    c. 图谱中源自受影响 chunk 的候选关系 → 退回专家审核队列
```

chunk_id 确定性公式不变：结构未变的条款重跑后 ID 相同（upsert 原位覆盖），结构修正导致消失的旧 ID 由"按 doc_version_id 范围 delete"清理孤儿，不依赖逐 ID 对账。

### 18.5 逃逸率指标与 golden set 反哺

- 新增治理指标：**逃逸率** = 当期 confirmed 工单涉及文档数 ÷ 当期 INDEXED 文档数，进质量看板，目标值 W2 起 < 1% ⚠
- **每一例确认逃逸强制反哺**：该文档（脱敏后）加入 golden set 成为回归样本；逃逸根因归类到 §18.1 类型并评审是否可规则化为新质检指标——闭环的学习机制，使关卡随逃逸样本累积持续收紧
- 逃逸案例库同时作为与甲方沟通"解析质量不可能 100%、但有发现-修复-收紧闭环"的证据材料，管理验收预期

### 18.6 本节新增项的属性声明

【成熟实践】：运行期用户反馈工单、按使用频次加权稽核、失效传播标记（不静默改结论）。【合成启发式 ⚠】：边缘通过带及其 ε 取值、双解析器仲裁的相似度阈值与覆盖范围、高危 token 词表与 0.95 局部阈值、context_quality_flag 机制、逃逸率 1% 目标。全部阈值随 V0 与 W1 基准批标定。

## 19. 异步富集层 S-E（CP-002，已批准）

> 位置：INDEXED **之后**异步运行（Temporal 独立低优先队列），富集失败降级为"无标签"，**不进质检阻断逻辑、不阻塞入库**。准入判断准则：被高频复用才前置（摊销合算）；依赖未冻结字典的标注必须可廉价重打（字典带 dict_version + 支持按版本增量重跑）。

### 19.1 E1 义务条款预打标（规则为主，零 LLM，全量）

- 道义词正则（应当/必须/不得/禁止/不予）+ obligation_keywords 词典命中 → `clause_tags.is_obligation`、`deontic_type`
- 同步正则预抽取**数值/期限/比例**字段（"N 个工作日""不超过 X 万元""X%"模式）入结构化列
- **期限单位归一化（V1.6 新增，CP-007 联动）** ⚠：所有期限表达强制归一到"日"存入 `norm_duration_days` 列，同时保留原文表达 `surface_duration`（standoff，不改 chunks.text）。归一规则：工作日按原文保留并标 `is_business_day=true`（区别于自然日）；月→30、季→90、半年→180、年→365 ⚠（金融监管文本常见近似口径，精确口径如"满 X 个月"按实际月份的边界由比对硬规则通道 §7.3 第⑤层兜底）。**动因**：甲方提问方式 18 问中"费用报销发票要在 3 个月内"与内规原文"九十日内"是同义不同表达，sparse 词法通道对面词"3 个月"匹配不到"九十日"——归一化后比对硬规则与查询数值过滤直接比 `norm_duration_days`，绕开表达差异。归一失败（无法解析的复合期限如"次年首个工作日"）保留原文不强行归一，标 `norm_status=unparsed` 计入看板
- 摊销收益（全富集层最高）：比对 Pass 2 第①步义务抽取从任务时移到入库时一次完成，单次比对省去 800–2,000 条现场抽取；第⑤层硬规则通道的数值比对直接查表（**含归一后期限**）。任务时保留现场兜底（防词典覆盖盲区）

### 19.2 E2 条款级涉及事项/责任部门/适用实体类型打标（LLM，字典约束，范围收敛）⚠

- 网关最轻量可用模型（CP-005-① 待确认；无小模型则用满血模型）、业务域/部门/**实体类型**字典约束输出、记录 dict_version；字典升版按版本增量重打
- **适用实体类型维度（V1.6 新增，CP-007 联动）**：在原 涉及事项 / 责任部门 两维之上增第三维 `entity_type[]`，约束于 `dict_entity_types` 枚举（C类营业部 / A类分支机构 / 期货IB / 投顾产品 / 上市公司控股子公司 等，初版由张老师评审，§16-7）。条文若显式限定适用对象（"C类营业部应当……"）则打标；无显式限定则留空（检索退化为不按实体过滤，不臆测）。**动因**：18 问中"C类营业部需要做机房出入登记吗""期货IB奖金发放对象"等实体型问句,检索阶段按 entity_type 过滤可避免召回适用于其他实体类型的条款
- 首期范围 = 内规全量（~4 万 chunk）+ 制度映射表命中外规（估 ~15 万 chunk）≈ 19 万次调用（成本口径由 GPU 日改为网关配额占用 ⚠，待 CP-005-①③ 确认后估算）；外规全量暂缓，待字典经一个比对周期验证稳定后放量
- **定位澄清（经分桶讨论修正）**：E2 服务的是**业务过滤**（比对按事项圈定、费用审计规则关联、查询按实体类型/事项圈定），不是检索精度优化——与检索机制类分桶（近重复坍缩、功能体裁桶等，另行提案）严格区分，不混用论证。V1.6 增的 entity_type 维度同属业务过滤范畴（圈定"这条规则适用于哪类实体"），论证不与检索精度混用
- 前置依赖：业务域/部门字典初版经张老师评审（待确认事项 §16-6）

### 19.3 E3 图谱优先级探针（规则统计，全量，近零成本）

- 逐文档计算关系密度指标：clause_references 内部引用密度、跨文档出/入度（R4 解析结果）、E1 义务条款密度 → 合成 `doc_graph_stats.graph_priority_score` ⚠
- 用途：图谱懒构建队列按分排序——同一业务域触发建图时高分文档先抽取，专家审核人时优先投向图谱收益最高的文档。即语料评估 V0 关系维度探针的**逐文档化**

### 19.4 E4 路由特征预留（明确的"不做"决策）

- E3 统计落 `doc_graph_stats`，作为二期若引入查询级图/向量动态路由的现成特征
- 一期路由维持**场景级静态**（查询不配图谱通道，理由与可证伪条件见主设计文档 §4.3 V1.1 修订）——此为决策而非遗漏，在此留痕

### 19.5 属性声明

【成熟实践】：富集与入库解耦的非阻塞设计、字典版本化增量重打。【合成启发式 ⚠】：E2 范围收敛策略与调用量估算、graph_priority_score 合成公式、E1 数值预抽取模式集、**E1 期限单位归一化规则（月→30/季→90 等近似口径，CP-007）**、**E2 实体类型打标维度（依赖 dict_entity_types 初版评审，CP-007）**。

## 21. 评测前置层（CP-006，已批准，T1–T6；T7 自检索探针按选项 B 移交未来分桶提案 CP-007）

在摄取期以近零边际成本采集 RAG 端到端评测所需的标注、基线与负样本。设计依据：摄取管线拥有两个事后无法复得的免费窗口——**构造即标注**（入库瞬间系统天然持有 ground truth）与**分批入库即免费规模实验**（1k→~6k→~21k→31k 四个台阶，窗口随全量入库永久关闭）。六个组件均挂富集层（§19）同款低优先异步队列，组件失败降级为"缺该项数据"，不进质检阻断逻辑；本提案纯采集评测数据，**不改变任何语料处理**。

### 21.1 T1 摄取即出题（评估集增殖）⚠

按 sub_type × biz_domain 分层抽样 2–3% 的 INDEXED 文档，异步生成问答对（轻量模型；问题 + 源 chunk_id 即检索 ground truth + 生成时刻元数据），写入评估集仓库（与 V0 评估集同 schema，add-only 独立表）。预期 W3 末累积 ≥2,000 问 ⚠，分布镜像语料构成。质量闸：抽样 5% 人工复核，问题合法率 ≥90% ⚠，不达标暂停生成并调 prompt。配额预算约 2–3 万次轻量调用（口径随 CP-005-① 落定）。

### 21.2 T2 批次检索冒烟测试（捕获"入库但检不到"）

批次 finalize 时对该批每份文档发射 1 条合成查询（优先复用 T1 产物），断言目标文档 hit@50 命中，且查询携带该文档应有的 perm_tag / status 过滤条件（**带权限语境的冒烟**）。失败 → 新错误段 E8xx（E801 检索不命中 / E802 过滤位异常），开质量工单但**不回退批次**（评测组件不拥有阻断权——故障可能在检索配置而非文档，回退是误伤）。验证对象是 §12.2 对账覆盖不到的一整类静默故障：权限位配错、status 滞留 staging、分区路由错、零向量。批次通过率 ≥99% ⚠ 纳入出口准则。

### 21.3 T3 规模台阶检查点（验证分区配额假设）⚠

固定 V0 评估集（重点内规题子集；T1 增殖题不混入，保证跨台阶可比性），在 S1=内规全量 / S2=外规首层 / S3=外规全量 / S4=全库 四个台阶各重跑，记录分桶 hit@k 与 MRR 曲线。判定：内规题 hit@10 相对 S1 基线下降 >5 个百分点 ⚠ → 当场调整分区配额（top25+25）或重排窗口并复跑确认。这是"外规 20 倍体量淹没内规结果"假设的唯一直接验证机会。判定阈值在 S1 基线建立后可校正一次。

### 21.4 T4 引用锚点自动回放验证

对每个 INDEXED chunk 程序化执行：按 page_start 取原件该页（±1 页）文本 → 断言 chunk 正文（剥离面包屑）可在其中定位（精确匹配；OCR 文档用编辑距离阈值 ⚠）。失败 → 质量工单（§18 通道复用）。批次通过率 ≥99% ⚠ 纳入出口准则。性质：把**依据验收**的核心断言（四级锚点真实可回放）从验收前人工抽查变为摄取期全量回归；§5.3 人工抽检的对页核对项相应缩减为复核性质。纯 CPU 成本。

### 21.5 T5 拒答负样本采集

从三类摄取记录免费生成"系统应触发依据不足"的测试问题：QUARANTINED / 退回甲方文档（知其存在而未入库）、pending_target 永久未解析被引文号（语料缺口清单复用）、样板条款跳过清单内容。每条负样本携带"为何不在库"元数据——拒答测试失败时可直接区分阈值问题 vs 越权内容入库。汇入 CP-004 对抗子集（其 ≥15% 占比的主要供给源）。

### 21.6 T6 分数分布基线采集

向量写入的同一时刻，按 chunk_type 落表桶内/桶间相似度分布分位数。用途：分桶分数校准与"依据不足"阈值标定的前置数据（校准机制本身随未来分桶提案 CP-007 另批）。零额外计算。

### 21.7 属性声明与范围边界

构造即标注、合成查询冒烟、回放式锚点校验为成熟实践方向；全部阈值数值（2–3% 抽样率、99% 通过率、5pp 台阶降幅、90% 问题合法率）为合成启发式 ⚠。**T7 自检索探针（难检索桶标记 + 定向增强）按选项 B 不纳入本层**：其下游会改变语料处理，归属分桶机制，与近重复坍缩、功能体裁桶等打包为 CP-007 候选；届时 T1 基础设施已就位，T7 实现成本仅剩判定逻辑。

## 22. 一期数据摄取全景的摄取侧承接（V1.6 新增，CP-008）

> 主设计文档《制度查询与制度比对智能体_RAG技术框架设计》V1.5 §13 从应用视角做了功能 5/6/7/8/9 的摄取边界声明。本节从**摄取侧**承接：P-MISC 因属文档摄取、与本管线"同一骨架、不同档案"哲学一致，在此给出完整 profile 定义；费用结构化数据、审计项目资料、模板三类不属于文档切块管线，在此给出**边界声明 + 交接指针**，详细设计另起独立文档。本节不改变制度/外规/QA/案例四类语料的任何现有处理。

### 22.1 边界总览

|---|---|---|---|
| 5 监督共享 | 部门输出文档（报告/清单/通报，无条款结构） | **是（新 profile P-MISC）** | §22.2 完整定义 |

### 22.2 P-MISC profile 完整定义（功能 5 监督共享）

监督共享文档是部门输出（合规事项 / 问责情况 / 监管函件 / 法律诉讼 / 风险事项），无"第X条"条款结构，强行走 P-INT/P-EXT 会在 §5.1 质检指标 1（条款覆盖率）批量失败、最终降级入库。P-MISC 是对管线骨架的合法 profile 变体：

| 配置项 | P-MISC 取值 | 与 P-INT/P-EXT 的差异说明 |
|---|---|---|
| 解析主通道 | DeepDoc 全要素（office/pdf 通道） | 同；扫描件仍走 OCR |
| 切块单元 | **段落/小节级**（按标题层级 + 自然段，目标 300–600 token） | 无条款树；§6.1 条款树构建**跳过**，§6.6/§6.7 指代预解析**跳过**（无条款引用语义） |
| 父子块 | 节级父块（有标题层级时） | 同骨架 |
| 版本链 | **无**（月度报送快照，不可变） | 无 revise_replace；按 `report_month` 管理，同月重报为新批次、旧批次标 superseded |
| 元数据 schema | 强化：来源部门 / 报送月份 / 文档类型枚举 / 标签数组 / 涉及业务域 | 无发文字号/生效日期；manifest 必填列重写（见下） |
| 质检方式 | **指标 1/2/3 禁用**（无条款），保留指标 4（页码锚点）/5（表格）/6（文本质量）/7（抽取充分性）；新增"必填元数据完整率" | §5.1 指标按 profile 选择性启用——指标表本就是 profile 维度的，此处声明 P-MISC 的启用子集 |
| 图谱纳入 | 否 | — |
| 入库优先级 | **独立月度轨**（不并入 W0–W3 制度/案例 bulk） | 月度固定导入,见 §22.2 末 |

**P-MISC manifest 必填列**（替代 §3.1 的制度类 manifest）：文件名 / 来源部门（合规法务/风险管理/审计中心）/ 报送月份(yyyy-mm) / 文档类型（合规事项/问责/监管函件/法律诉讼/风险事项/其他）/ 标签 / 密级 / 涉及业务域。无发文字号、无版本说明列。

**检索隔离（硬边界）**：P-MISC 文档入独立 partition `corpus_type=supervision_share`，**默认不进制度查询智能体的依据查询检索范围**。功能 5 的报告生成是独立后台任务型工作流，从该 partition 按 report_month 拉取汇总，不经查询智能体路由。理由：监督共享是"部门工作信息"，不是"判断依据"，混入依据查询会污染引用追溯的"制度来源"语义。

**引用追溯降级**：P-MISC 不可达条款级，引用仅到文档级+段落级，依据展示处标识"非制度类来源（监督共享信息）"，与制度类四级锚点显式区分。

**月度导入轨**：P-MISC 不参与 §13.2 的 W0–W3 一次性 bulk，而是上线后每月固定批次。复用同一 IngestBatchWorkflow（§11.1），manifest/profile 切 P-MISC 即可；缺失文件说明（某部门某月未报）记入批次质量报告，对应启动会"记录导入月份、导入状态、缺失文件说明，不做催办"的边界口径。

### 22.3 费用审计数据（功能 6/7，边界声明）

费用审计是**混合摄取**，两条通道形态差异大，须拆分：

- **发票图像 / 附件 PDF**：走本管线 OCR 通道（类比 P-CASE 的扫描件处理），独立 partition `corpus_type=expense_doc`；OCR 重点字段为金额/日期/对方主体/发票号；可复用 §18.2 高危 token 复核（金额数字防 OCR 错位）。
- **费用明细 / 预算 / 审批流水**：**不走文档切块管线**。这是结构化业务数据（行记录），应入独立业务表族（建议 `expense_records` / `budget_lines` / `approval_records`，schema 由费用审计独立文档设计），通过 §4.1 的 xlsx 直读路径接入（一期"接口预留+文件导入兜底"口径），查询走 SQL 而非向量检索。**这是摄取-业务表分离的架构边界**：本管线只负责把它们从 Excel/CSV 落到业务表，不做 chunk 化、不进 audit_corpus。
- **报销标准（属内规）**：走现 pipeline P-INT，作为费用审计规则引擎的规则参考来源。

**交接**：费用结构化数据的业务表 schema、规则引擎与 LLM（OCR 字段补全 + 违规原因解释）的分工，由《费用智能化审计_技术框架设计》承接。本管线交付边界 = Excel 直读 + 业务表落地。

### 22.4 审计报告项目资料（功能 8，边界声明）

被审计单位资料有**项目级生命周期**（项目期一次性使用、项目结束归档/删除），与知识库语料的长期持有形态不同：

- 文档解析复用本管线（解析→质检→切块），但**引入 project_id 维度**，入独立 partition `corpus_type=audit_project`，按 project_id 分组；项目结束触发归档动作（移出在线检索）。
- **检索隔离**：项目资料强制 perm_tag + project_id 过滤，不与知识库主检索通道混合；**不进入依据查询结果**——项目资料是"被审计对象的数据/事实素材"，不是"判断依据"。审计报告生成时与制度知识库联合检索（项目资料作事实、制度作依据），但这是报告生成工作流的事，不是本管线的事。

**交接**：项目生命周期管理、归档策略、报告生成的联合检索逻辑，由《审计报告智能化_技术框架设计》承接。本管线交付边界 = 带 project_id 的文档解析入库。

### 22.5 模板库（功能 9，边界声明）

**模板不属于语料**。模板是带占位符的可执行结构，与 chunks/cases 表族无关，不进本管线任何环节：

- 存储：MinIO（模板原件）+ PG（模板元数据：template_id / 名称 / 适用场景 / 字段映射 schema / 章节配置 / version / 默认标记 / 权限点 / 使用日志）。
- 调用：场景 4（审计规范生成）、场景 8（审计报告生成）、查询智能体导出（主文档 §6.7）通过模板服务调用，传入 placeholder values → 渲染 → 输出 docx/xlsx。
- 不进入任何检索路径。

**交接**：由《模板库与模板管理_技术框架设计》承接。

### 22.6 摄取侧设计文档归属与本管线的联动改动
