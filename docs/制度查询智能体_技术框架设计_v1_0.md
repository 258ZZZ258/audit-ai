# 制度查询智能体：技术框架设计方案（V1.0）

> 适用项目：东方证券审计大模型系统一期（功能 1 制度查询智能体）
> 覆盖范围：查询理解前端 → 意图路由 → 混合检索/重排 → 引用约束生成 → 多模型复核 → 输出契约 的**运行时全链路**（即《制度查询与制度比对智能体_RAG技术框架设计》V1.5 §5–§6 制度查询部分的专项展开与下沉）
> 上游依赖：《文档处理与语料库构建_技术框架设计》V1.6 提供的语料、PostgreSQL 权威条款表、Milvus 检索投影、富集层标注（E1 义务/期限归一化、E2 涉及事项/实体类型、clause_references 指代表）与四级引用锚点。本文档**消费**这些资产，不重述其构建过程
> 语料规模：内规 ≈ 1,000 件，外规（含监管问答）≈ 20,000 件，监管案例 ≈ 10,000 件，估算 60–95 万 chunk
> 设计依据：项目启动会修订版（2026-06）+ 甲方提问方式 18 问与应用场景样本（2026-06，驱动 §3 查询理解前端与 §6.5/§6.6 判定/统计型路由）+ 既有架构决策（LangGraph / bge-m3 混合检索 / Milvus 2.4 / 必要性覆盖范式 / CP-007 查询增强一揽子）
> 文档性质声明：【标准】= 行业成熟实践；⚠ = 合成启发式/工程估算，须实测标定，对甲方陈述前经张翼飞复核
> 边界声明：本文档**不含**制度比对智能体（功能 2，见主设计文档 §7，另文展开），仅在共享机制与边界处提及；专业判断/概念辨析/因果推理类任务**不在承诺范围**（见 §0.3）；摄取与语料构建不在本文展开（见 V1.6）

## 0. 设计目标与硬约束

### 0.1 目标与约束矩阵

| # | 目标/约束 | 来源 | 本设计落位 |
|---|---|---|---|
| 1 | 依据不足时明确提示，**绝不生成无来源条款，绝不出"违规/合规"裸结论** | 启动会项目核心目标 + 项目红线 | §7.1 引用 ID 注入 + §8 覆盖感知拒答 + 判定型框定 |
| 2 | 每条回答的依据可**四级回溯**（条款→文档→页码→版本） | 依据验收 | §7.3 引用锚点由 PG 权威条款表回查注入 |
| 3 | 制度查询为**纯 chatbox**，不出预设勾选条件 | 甲方明确否决复选框（已验证 UX 硬约束） | §3.4 多轮归并 + §6.7 澄清纯对话 |
| 4 | 默认**不引用废止制度** | 依据验收隐含前提 | §5.3 `status=现行` 强过滤位 |
| 5 | 数据流**单向只读**，不回写任何源系统 | 项目红线 | 全链路无写回；知识库更新仅人工导入（V1.6） |
| 6 | 权限**前置过滤** + 操作日志全程留痕 | 权限验收 / 安全验收 | §5.3 检索前置 perm_tag 过滤 + §9.3 Casbin + 操作日志 |
| 7 | 多模型复核 / 敏感词过滤 / 生成式 AI 内容标识 | 非功能验收 | §9 横切能力 |
| 8 | 平均 QPS 50，长任务异步 + 状态展示 | 性能验收 | §12 容量测算（查询为同步流式，QPS 主体为轻请求；长任务属比对智能体） |
| 9 | OpenAI API 兼容模型网关 + MCP + SKILL | 非功能验收 | §9.1 网关与模型矩阵 |
| 10 | 前端 Vue + genesis-ui（甲方组件库） | 组织架构约束 | §10 输出契约前端无关化（后端只交付 JSON） |

### 0.2 一句话架构

**查询理解前端**把口语问句翻译进检索空间、**八路意图路由**按问句形态分流、**引用 ID 注入式生成**从机制上杜绝编造、**覆盖感知拒答与判定型框定**守住"无依据结论"红线——四根支柱之上复用平台层 RAG 检索 / Agent 编排 / 引用追溯能力，前端仅消费统一 JSON 契约。

### 0.3 承诺范围外（边界声明，须张翼飞复核）

| 不做项 | 原因 | 建议去向 |
|---|---|---|
| 专业判断 / 概念辨析（如"增发是否等于换股"） | 需多步 plan→retrieve→reason→re-retrieve→synthesize 循环，产物是**推理链**而非引用定位，复核是**推理有效性**而非引用真实性——与"检索+引用约束生成"存在结构性差异 | 独立立项或挂载场景 4（审计规范生成）扩展 |
| 因果方向辨析（如"估值-会计估计变更因果方向"） | 同上 | 同上 |
| 监管认定 / 审计要点合成 | 同上 | 同上 |
| 案例分析能力 | §6.3 案例桥接检索属**查询增强通道**，仅做"口语↔法言"翻译与依据附挂，**不构成案例分析** | 场景 3（监管案例舆情分析，后台任务型，另文） |
| 制度比对（必要性覆盖、四态分类、差错点判定） | 功能 2，后台任务型，长耗时批量比对 | 主设计文档 §7（另文展开） |

## 1. 总体架构与运行时状态机

### 1.1 在"1 平台 + 多形态应用"中的定位

制度查询智能体是 **Chatbox 型 / 同步流式**应用，复用平台能力层与基础设施层，**不自建私有检索通道**（schema 变更遵循 add-only 约束，与比对智能体共用同一套解析管线、Chunk 仓库、Embedding 服务、Milvus 集合、引用追溯 schema、模型工厂）。

│   ┌──────────────────────────────────────────────┐            │
│   │ 制度查询智能体（Chatbox 型 / 同步流式）           │ ← 本文档    │
│   │   查询理解前端 → 八路路由 → 检索/重排 →           │            │
│   │   引用约束生成 → 多模型复核 → 流式输出            │            │
│   └───────────────────┬──────────────────────────┘            │
├───────────────────────┼────────────────────────────────────────┤

├──────────────────────────────────────────────────────────────┤

### 1.2 运行时状态机（LangGraph）

查询智能体的"状态机"是一次问答的运行时编排图（对应 V1.6 摄取管线的文档级状态机）。节点为 LangGraph 节点，边为条件路由。

```
[用户输入]
  → N0 多轮上下文归并（指代消解 / 补全省略的制度名）
  → 查询理解前端（CP-007）：
       N1 HyDE 改写（口语 → 假设性法言）         [默认 on，边界场景 off 待 V0 标定]
       N2 业务事项分类（E2 事项 + dict_scenario_terms 桥接 + entity_type 抽取）
       N3 问题分解（复合问句拆子查询；单跳直通）
  → N4 意图路由（8 路）
  ├─ R1 依据查询 ──→ 混合检索(内规∥外规分区并行) ∥ 案例桥接通道(§6.3)
  │                   → 重排 → status/perm_tag/entity_type 过滤 → N5 充分性自检
  │                   ├─ 充分 → N6 引用约束生成(流式) → 案例附挂 → 输出+导出
  │                   └─ 不足 → N7 覆盖感知拒答(§8)
  ├─ R2 变更查询 ──→ 制度定位 → 版本链回查(PG) → 条款级 diff + 修订说明/背景归并
  ├─ R3 相似案例 ──→ case 分区检索 → 要素表回填 → 案例卡片输出
  ├─ R4 多文档列举 → E1 义务标签 ∩ E2 事项 ∩ biz_domain ∩ entity_type（枚举模式：高 k 不激进截断）
  ├─ R5 判定型 ───→ 案例桥接为主 + 依据条款 + 构成要件框定 → 多模型复核 → 三段式输出
  ├─ R6 统计型 ───→ 维度抽取 → cases 参数化 SQL 聚合 → 表格化输出
  ├─ R7 需澄清 ───→ 单问题澄清（纯对话，不出复选框）
  └─ R8 兜底 ─────→ 礼貌拒答（超出审计制度域）
```

### 1.3 设计上的四个关键取舍

1. **查询理解前端 vs 裸检索**：每次查询增 ~2 次轻量调用（HyDE + 事项分类），换取术语断层问句（口语描述命中法言条款）的召回提升。这是一个带退路的工程押注——HyDE 默认 on / off 待 §13 V0 实验轨第 5 组 A/B 实测；若仅术语断层桶有提升，可降级为"分类器识别为行为咨询型 + 高断层时才触发"，配额减半。
2. **八路路由 vs 单一 RAG**：六类问句形态（依据/变更/案例/列举/判定/统计）的**产品形态本就不同**——尤其判定型问句若混入依据查询路由，主答模型容易顺势给出裸结论。用路由从**形态上**隔离裸结论风险，是架构决策而非过度设计（与 §0.1-#1 红线直接挂钩）。
3. **引用 ID 注入 vs 自由生成**：引用 ID 由系统注入，LLM **只做选择不做生成**，从机制上杜绝编造发文字号/条号。即使 HyDE 编出貌似合理的错误法言，也因为最终答案只能引用检索上下文中带 `clause_id` 的内容而被兜住（§7.1）。
4. **覆盖感知拒答 vs 分数阈值拒答**：对存在性问句（"有没有规定…""违不违规""需不需要"），"top1 分数 < 阈值"系统性失效——检索为空时系统区分不了"确实无禁止性规定"和"我没召回到"。改判据为"是否在相关业务事项分区内穷尽检索过"（§8.1）。

## 2. 查询侧消费的语料资产与检索视图

本文档不构建语料，只声明查询智能体**从 V1.6 摄取产物中消费什么、怎么消费**。

### 2.1 资产消费映射

| 资产（V1.6 产出） | 查询侧消费方式 | 服务的路由 |
|---|---|---|
| `audit_corpus`（Milvus，partition key = corpus_type） | 混合检索主通道；internal/external 分区**并行检索**（§5.2） | R1/R4/R5 |
| Milvus 标量：`status` | 默认强过滤 `=effective`；历史版本问句显式放开（§5.3） | 全部 |
| Milvus 标量：`perm_tag` | **检索前置**权限过滤（filter 而非后裁剪，§5.3） | 全部 |
| Milvus 标量：`entity_type[]`（CP-007） | 问句命中实体类型（如"C 类营业部"）时强制过滤 | R1/R4/R5 |
| Milvus 标量：`biz_domain[]` / `issuer_level` / `effective_date` | 事项圈定 / 义务来源优先级 / 时间窗 | R1/R4/R6 |
| Milvus 标量：`chunk_type`（clause/table/qa/case_summary/case_section） | 列举型偏好 clause、相似案例命中 case_summary | R3/R4 |
| PG `chunks`（**全文不截断 + breadcrumb + page_start/end + parent_chunk_id**） | **引用展示与父块供证的权威源**——回答中条款原文/页码/版本一律从此回查，不用 Milvus 截断文本 | R1/R5（§7.3） |
| PG `doc_versions` + `revision_notes` | 变更查询的版本链回查与"修订原因"权威源 | R2（§6.2） |
| PG `cases`（处罚机构/对象/事由/**引用外规条款**/金额/日期） | 案例桥接精确反查 + 相似案例卡片 + 统计型 SQL 聚合 | R3/R5/R6 |
| PG `clause_references`（standoff 指代表，R1–R4） | 多跳关联的**确定性拦截**（"前款""依照第 X 条"免检索直接查表）——查询一期不配图谱的支撑证据之一 | R1/R2 |
| `clause_tags`（E1：is_obligation/deontic_type/norm_duration_days；E2：涉及事项/责任部门/entity_type[]） | 列举型义务过滤、覆盖感知拒答的"事项分区穷尽"判据、期限数值过滤绕开"3 个月 vs 九十日"表达差异 | R4/R5/§8 |
| `dict_scenario_terms`（情景术语桥接词典，CP-007） | 业务事项分类前置匹配 + sparse 通道扩展（"代客理财→全权委托/受托理财"） | N2/§5.4 |
| `dict_intent_routes`（路由训练样例库，CP-007） | 意图路由器持续优化的样例源（含金总真实查询日志 + 18 问/应用场景冷启动种子） | N4 |
| `dict_entity_types`（适用实体类型枚举，CP-007） | entity_type 抽取与过滤的约束词表 | N2/§5.3 |

### 2.2 三类语料在查询中的角色

| 语料 | 在查询中的角色 |
|---|---|
| 内规 (1k) | 查询主体；最高保真度，条款级结构化 + 全量人工抽检（依据回显权威） |
| 外规 + 监管问答 (20k) | 查询主体；义务来源方；体量为内规 20 倍，检索须分区配额防淹没（§5.2） |
| 监管案例 (10k) | **桥接通道入口 + 附挂输出**（CP-007，§6.3）；统计型路由的 SQL 数据源（§6.6） |

### 2.3 规模与单机结论（引用 V1.5 §2.2）

60–95 万 chunk、dense 1024d 约 4 GB，Milvus 单机 standalone（HNSW，×1.5 系数）**16–32 GB 内存可承载**，一期不需分布式集群，按 `corpus_type` 做 partition key 分区即可 ⚠（须 V0 用真实 200 件抽样平均 chunk 数回归修正；若外规多含法律全文，chunk 总量可能上探 150 万+、内存上调至 64 GB，仍不改单机结论）。

## 3. 查询理解前端（CP-007）

查询理解前端把"用户怎么说"翻译成"检索系统怎么找"，是 CP-007 的核心增量。三节点串联，全部可降级（失败回落原始问句/默认分支，不阻断查询）。

### 3.1 N1 HyDE 改写

| 项 | 设计 |
|---|---|
| 任务 | 口语问句 → 1–2 句**假设性法言条款**，与原始问句一同送入检索 dense/sparse 通道 |
| 模型 | 网关轻量模型（CP-005-① 待确认，无小模型则满血承担，§9.1） |
| 失败处理 | 回落原始问句（不阻断） |
| 污染兜底 | HyDE 编出貌似合理的错误法言**不污染最终答案**——引用 ID 注入（§7.1）保证答案只能引用真实检索上下文 |
| on/off 决策 | 进 §13 V0 实验轨第 5 组 A/B；hit@10 整体提升 >5pp ⚠ + 边界无负面 → 默认 on；仅术语断层桶提升 → 桶触发（配额减半）；无提升/污染 → 砍掉回退混合检索 + 情景词典 |

【标准】HyDE（Hypothetical Document Embeddings，Gao et al., 2022/2023）。⚠ on/off 默认值与配额折算待实测。

### 3.2 N2 业务事项分类器

| 项 | 设计 |
|---|---|
| 任务 | 输出三组标签：① **场景类型**（行为合规咨询 / 概念定义 / 列举枚举 / 变更追溯 / 统计聚合）；② **涉及事项**（映射 E2 涉及事项字典）；③ **entity_type** 抽取（命中 dict_entity_types 枚举，如"C 类营业部"） |
| 模型 | 网关轻量分类模型 |
| 直接作用 | 场景类型标签 → 驱动 §4 路由分支选择；涉及事项 → 覆盖感知拒答的"事项分区"（§8.1）；entity_type → §5.3 检索强过滤 |
| 桥接 | 前置匹配 `dict_scenario_terms`，把口语词映射为法言词（"二维码介绍开户"→"违规招揽客户/居间介绍"），服务 sparse 通道扩展 |

### 3.3 N3 问题分解

仅对**显式复合问句**触发（如"私募权益投顾同时管偏股+偏债是否违规"含多个子约束），拆为子查询后并行检索再综合。单跳问句直通，不引入额外延迟。**关键边界**：不进入 plan→retrieve→reason→re-retrieve→synthesize 的 agentic 循环（§0.3）——分解只做一次性子查询拆分，不做迭代推理。

### 3.4 N0 多轮上下文归并

| 项 | 设计 |
|---|---|
| 指代消解 | 多轮中"它/该制度/上面那条"等指代，结合会话上下文补全 |
| 省略补全 | 补全省略的制度名/业务域（接上一轮主题） |
| 硬约束 | **纯 chatbox，不出预设勾选条件**——甲方明确否决过复选框（预选会导致漏选）。澄清走纯对话（§6.7），不渲染任何条件复选框 |

> 全链路开销说明（CP-007）：每次查询新增 ~2 次轻量调用（HyDE + 事项分类），问题分解仅复合问句触发（日常占比 <30% 估计 ⚠）。无轻量小模型时退化为满血承担，QPS 50 下日新增满血调用约 7,500 次/小时，须在网关配额申请中预留（§12）。

## 4. 意图路由（八路）

### 4.1 路由器

- 轻量分类模型，输入 = 原始问句 + N2 场景类型标签，输出 = 八路标签 + 置信度
- 训练样例库 `dict_intent_routes`：**这是真实查询日志的第一个落点**；金总提供的合规部门真实查询日志 + 18 问 + 应用场景 5 问应直接灌入作为冷启动种子，线上日志到位后持续校正分布

### 4.2 八路定义

| 路由 | 触发（N2 场景类型 + 问句特征） | 主路径 | 工作流 |
|---|---|---|---|
| R1 依据查询 | "费用报销发票 3 个月规定在哪" 等 what/where 型 | 混合检索 → 重排 → 充分性自检 → 引用约束生成 | §6.1 |
| R2 变更查询 | 含"变更/修订/旧版/什么时候改的"信号 | 版本链回查 + 条款 diff + 修订说明 | §6.2 |
| R3 相似案例 | "有没有类似处罚案例" | case 分区检索 → 要素回填 → 卡片 | §6.3 |
| R4 多文档列举 | "哪些制度规定了 X""列出所有关于 Y 的要求" | E1∩E2∩biz_domain∩entity_type，枚举模式高 k | §6.4 |
| R5 判定型 | 行为合规/违规咨询型（18 问约 1/3：二维码开户、见底到顶、专用章保管、机房出入登记…） | 桥接 + 依据 + **构成要件框定**，三段式 | §6.5 |
| R6 统计型 | 聚合统计型（"哪些板块处罚高发""2024 以来期货监管处罚有哪些"） | 维度抽取 → cases 参数化 SQL | §6.6 |
| R7 需澄清 | 路由置信度低 / 问句歧义 / 关键约束缺失 | 单问题澄清（纯对话） | §6.7 |
| R8 兜底拒答 | 超出审计制度域 | 礼貌拒答 | §6.8 |

### 4.3 置信度与回退

- 路由置信度低于阈值 ⚠ → 走 R7 澄清（单问题），不强行分流
- 多标签命中（如既像判定型又像列举型）→ 优先级规则：判定型 > 统计型 > 变更 > 列举 > 依据（判定型优先以确保裸结论风险被路由形态拦住）⚠，优先级序列待 V0 与真实日志校正

## 5. 检索与重排

按既定全管线口径逐环节给出。【标准】= 成熟实践；⚠ = 工程判断。

| 环节 | 选型 | 理由 / 备注 |
|---|---|---|
| 5.1 混合检索 | Hybrid（dense bge-m3 + sparse bge-m3 lexical）+ Milvus 2.4 原生 RRF 融合 | 【标准】混合检索对法规专有名词（发文字号、罚则编号）精确命中显著优于纯稠密；sparse 输出免单独 BM25 服务 |
| 5.2 分区并行配额 | 内规 / 外规**各打各自 partition**，各取 top25 重排合并 ⚠ | 避免外规 20 倍体量在统一检索中淹没内规结果——分区配额是对语料不均衡的直接对策。配额由 V0 实验轨 T3 台阶检查校正 |
| 5.3 强制过滤位 | `status=effective`（默认）+ `perm_tag`（前置）+ `entity_type`（条件性）+ `biz_domain` | status 决定不引用废止制度（依据验收隐含前提）；权限**检索阶段完成**禁止后裁剪；entity_type 命中实体型问句时强制过滤，避免召回适用于其他实体的条款 |
| 5.4 sparse 精确通道 | 发文字号 / 制度全名走 sparse 权重提升；语义问句 dense 主导；`dict_scenario_terms` 扩展 sparse 命中面 | 精确编号问句 vs 语义问句的差异化权重 ⚠ |
| 5.5 重排 | bge-reranker-v2-m3，top50 → top8 ⚠ | 既定默认；top-k 待 V0 标定 |
| 5.6 父子块供证 | 命中子块（条款级）→ 回查父块（节级，PG）供 LLM 完整上下文 | 【标准】"小块检索、大块供证"——引用精度与上下文完整性兼得 |
| 5.7 充分性自检 | 升级为"覆盖语境判据"（非 top1 分数阈值） | 详见 §8.1 |

> 默认 `status=effective`；用户**显式问历史版本**时放开过滤，并在回答中标注"该版本已废止/被替代"。

## 6. 八路路由工作流详解

### 6.1 R1 依据查询

```
问句 → 混合检索(内规∥外规分区并行,各 top25) ∥ 案例桥接通道(§6.3,仅行为咨询型触发)
  → 重排 top50→top8 → status/perm_tag/entity_type 过滤
  → 充分性自检(§8.1)
  ├─ 充分 → 引用约束生成(流式,§7) → 案例附挂(§6.3 附挂通道) → 输出 + 导出按钮
  └─ 不足 → 覆盖感知拒答(§8.2)
```

检索细节：发文字号/制度全名走 sparse 精确通道权重提升；语义问句 dense 主导；命中实体类型则 entity_type 强过滤。

### 6.2 R2 变更查询

启动会要求输出"版本、变更内容、**变更原因、背景情况**"。变更内容可由条款级 diff 机器产出；**变更原因和背景无法从正文 diff 推断**，权威来源是修订说明和发布通知。

| 输出栏 | 来源 | 缺失处理 |
|---|---|---|
| 版本 | `doc_versions` 版本链 | — |
| 变更内容 | 条款级 diff（机器产出，新旧版对齐） | — |
| 变更原因 | `revision_notes`（修订说明全文 + 按条目切分） | 缺失 → 明确显示"修订说明未提供，无法给出官方修订原因"，**不允许 LLM 推测** |
| 背景情况 | 允许附挂同期监管动态案例（cases，按日期窗口检索） | 必须标注"同期监管背景参考"而非官方原因 |

修订说明条目与机器 diff 的对齐由 LLM 辅助 + 置信度标注 ⚠，对齐失败的条目保留原文不强行结构化。

### 6.3 R3 相似案例 + 案例桥接双通道（CP-007 核心升级）

**核心洞察**：处罚案例正文用接近口语的事实描述（"当事人通过微信发送开户推广二维码……"），同时 `cases.引用外规条款` 字段已结构化指向所违反的外规条款。对**行为合规咨询型问句**（用户用具体业务场景的口语描述、命中的却是法言法语条款），案例正文与口语问句词面高度重合，可作为"口语↔法言"的术语桥梁——案例从 V1.4 的"事后装饰"升级为"检索路径本身"。

| 通道 | 用途 | 工作流位置 |
|---|---|---|
| **桥接通道**（CP-007 新增） | 行为咨询型问句的**检索入口**：口语问句先打 case 分区 → 命中案例 → 顺 `cases.引用外规条款` 精确跳转至外规条款 → 拿到被违反的外规条款原文送入主答上下文 | R1/R5 主路径前置（与混合检索并行） |
| **附挂通道**（原 V1.4） | 依据查询回答完成后追加：精确反查（外规条款→cases）∪ 语义检索 case 分区，取 top3 卡片附在回答尾部 | R1 主路径后置 |

**触发条件（仅桥接通道）**：N2 判定问句类型为"行为合规/违规咨询型"时触发；定义型/列举型/变更型/统计型不走桥接（避免过度推销）。

**适用边界（明确不做）**：概念判断型（"增发是否等于换股"，无对口案例）、因果辨析型（超出范围，§0.3）、纯依据查询（"费用报销发票 3 个月规定在哪"，精确依据已足）均不走桥接。

> 与比对智能体第 6 层的关系：本通道与比对 §7.3 第 6 层"案例反推补充检测"复用同一 `cases.引用外规条款` 精确反查机制，无新增工程开销，仅查询侧消费路径不同。

R3 纯"相似案例"查询（用户直接问"有没有类似案例"）则走 case 分区语义检索（主命中面 `chunk_type=case_summary`）→ 要素表回填 → 案例卡片输出（标题 + 处罚机构 + 日期 + 事由 + 引用外规条款）。

### 6.4 R4 多文档列举（枚举模式）

```
问句 → E1 义务标签 ∩ E2 涉及事项 ∩ biz_domain ∩ entity_type 过滤
  → 枚举模式：高 k 检索，不激进截断（确保召回完整性，CP-007）
  → 去重 + 按 doc/clause 聚合 → 列表化输出（每项带四级锚点）
```

**枚举模式 vs 默认检索的差异**：默认检索追求 top8 精排；列举型追求**召回完整性**，重排窗口放大、不做激进截断。**有效边界依赖 E2 对外规的覆盖范围**（V1.6 §19.2）：若 E2 仅打标内规 + 映射外规子集，则"列出所有外规中关于 X 的要求"无法保证完整——该边界须明确声明（§15-③），不可对甲方承诺穷举外规。

### 6.5 R5 判定型路由（CP-007 新增，红线落实核心）

**触发**：N2 判定为"行为合规/违规咨询型"（18 问约 1/3）。

```
判定型问句
 → 桥接通道(§6.3)：案例反查 → 外规条款定位
 → 混合检索(内规+外规)补充候选条款
 → 重排聚合 → 构成要件提取(LLM 从命中条款抽取"适用前提/对象/行为类型")
 → 三段式输出(prompt 模板硬约束结构)：
    ① 相关依据条款(四级锚点)
    ② 构成要件框定(结构化展示该条款适用边界："该规则适用前提为 X/Y/Z；本问句涉及的事实需具备…")
    ③ "AI 辅助判断,建议人工复核"标识 + 多模型复核标记
 → 引用约束 prompt 强制"不出违规/合规裸结论"
 → 多模型复核校验试探性表述(§9.2)
 → 输出(route_type=judgmental,前端差异化渲染人工复核框)
```

**为什么不走 R1 依据查询**：R1 产品形态是"问什么答什么"，judgmental 问句若走 R1，主答模型容易顺势给裸结论（即使引用条款是真的）。R5 通过 prompt 模板硬约束输出结构，从**形态上**杜绝裸结论。

**关键设计点**：
- **构成要件框定**而非"我也不知道"——案例桥接已把最接近的依据捞到，构成要件框定是把"判断这一步"留给人。这是工程能做的极限，也是红线允许的边界。
- **不引入多步推理循环**——单轮检索 + 结构化输出，不进 agentic 循环（§0.3）。
- **与比对智能体边界**：比对"差错点"是有明确依据（双向交叉验证或硬规则触发）的差异认定；R5 处理的是"用户不知道规则、想知道行为是否触线"的咨询场景，无既有结论，二者不重复。

> **待确认（P0，§15-④）**：R5"不出裸结论"的产品形态需经张益确认是否满足业务期望——这直接关系验收口径与红线落实。

### 6.6 R6 统计型路由（CP-007 新增）

```
统计型问句
 → 维度抽取(LLM 识别聚合维度:板块/事由/时间窗/处罚机构/对象类型)
 → 参数化 SQL 生成(基于 cases 表结构化字段;SQL 模板预定义,LLM 仅填参数,防注入)
 → SQL 执行 → 结果表格化输出(按聚合值排序)
 → 可选:附 top N 案例链接供下钻
```

**关键设计点**：不走向量检索（统计聚合 top-k 给不出答案，必须 SQL）；前提依赖 cases 表 §核心字段完整率 ≥90% 闸 + 违规类别字典经张老师评审落地；与场景 5（监管案例舆情分析，后台任务型周期报告）边界——R6 是查询智能体内的**即问即答轻量分析通道**，数据源同为 cases 表，无重复。

| 问句示例 | 输出 |
|---|---|
| "2024 年以来期货监管处罚有哪些" | 按发布日期降序的案例列表（标题 + 机构 + 日期 + 类型） |
| "哪些板块处罚高发" | 按违规事由分类聚合的板块计数表，降序 |

### 6.7 R7 需澄清

单问题澄清，**纯对话**（不出复选框，§3.4 硬约束）。一次只问一个最关键的缺失约束（如"您问的是现行版本还是某个历史版本？"），用户回答后回到 N0 重新归并。

### 6.8 R8 兜底拒答

超出审计制度域的问句礼貌拒答，说明系统能力边界，引导回制度查询场景。

## 7. 生成与引用约束

### 7.1 引用 ID 注入式生成（机制性反编造）

| 机制 | 设计 |
|---|---|
| 上下文构造 | 检索/重排后的每个候选块在送入 prompt 时**带系统注入的 `clause_id`** |
| prompt 强约束 | 只允许引用上下文中带 `clause_id` 的内容；引用必须标 `clause_id`，禁止凭记忆生成发文字号/条号 |
| 本质 | LLM 做**选择**不做**生成**——从机制上杜绝编造，而非靠模型"自觉" |

【标准】引用 ID 注入式生成约束。

### 7.2 流式输出

主答模型 Qwen3.5（甲方网关满血，CP-005）流式生成，首 token < 3s ⚠ 改善体感（端到端 P95 < 12s，§12）。

### 7.3 引用四级锚点回查

回答中展示的条款原文、页码、版本、状态**一律从 PG `chunks` / `doc_versions` 回查**（不用 Milvus 截断文本），避免 chunk 截断导致引用失真。四级锚点 = `clause_id → doc_version_id → page_start/end → version+status`。

### 7.4 prompt 模板分路由约束

不同路由用不同 prompt 模板硬约束输出结构：R1 依据查询"问什么答什么 + 引用"；R5 判定型强制三段式 + 禁裸结论（§6.5）；R2 变更查询按"版本/内容/原因/背景"四栏 + 缺失明示。

## 8. 覆盖感知拒答与判定型框定（红线落实）

V1.4 的依据不足策略基于"top1 分数阈值"，对**存在性问句**系统性失效。CP-007 升级判据如下。

### 8.1 分数阈值 → 覆盖语境判据

```
问句 → N2 映射到 E2 涉及事项(如 事项=投顾业务)
  → 在该事项分区内执行高召回检索(top50,所有候选 E1 义务条款)
  ├─ 有命中 → 充分 → 引用约束生成
  └─ 仍无命中 → 覆盖感知拒答话术(§8.2)
```

判据从"top1 分数 < 阈值"升级为"**是否在相关业务事项分区内穷尽检索过**"。

### 8.2 覆盖感知拒答话术（替代旧"依据不足"）

> "在【投顾业务】现行制度中**未检索到对该行为的明确禁止性规定**，但不等于无规定（可能涉及其他事项分类或穿插条款）。以下 N 条最接近供人工核实：……"

附"已穷尽的事项分区 + 最接近的 N 条"，使拒答可解释（区别于无信息量的"依据不足"）。

### 8.3 判定型框定三段式（防"无依据结论"红线踩线）

R5 输出绝不出"违规/合规"裸结论，强制三段式：① 相关依据条款（四级锚点）+ ② 构成要件框定 + ③ "AI 辅助判断，建议人工复核"标识。多模型复核（§9.2）专门校验任何试探性表述（"可能违反""疑似违规"）是否被引用条款支持；不支持的降级为"待核实"。

### 8.4 与三条红线的对应

| 红线 | V1.4 旧策略风险 | V1.0（CP-007）落实 |
|---|---|---|
| 不生成无依据结论 | 直接给"违规/合规"判定 → 踩线 | R5 仅输出"依据 + 构成要件"，不出裸结论 |
| 拒答应可解释 | "依据不足"无信息量 | 覆盖感知拒答附"已穷尽事项分区 + 最接近 N 条" |
| 引用真实性 | 多模型复核仅覆盖引用真实性 | 复核同步检查"试探性表述是否被引用支持" |

## 9. 横切能力（启动会非功能验收落位）

### 9.1 模型网关与模型矩阵（CP-005）

所有 LLM/Embedding/Rerank 调用经甲方模型网关（OpenAI API 兼容）。**算力责任方为甲方**（已本地部署全参数满血模型），我方测算对象从 GPU 采购转为网关并发配额（§12）。

| 任务 | 模型 | 依据 |
|---|---|---|
| 查询主答 | Qwen3.5（甲方本地满血） | 复杂推理主力 |
| 多模型复核 / 长上下文 | Kimi 2.5（甲方本地满血） | 异构 + 长上下文适配 ⚠ |
| 意图路由 / HyDE / 业务事项分类 / 问题分解 / 维度抽取等轻任务 | 网关轻量模型（可用性待对接会确认 CP-005-①） | 无小模型则满血承担——分层动机为省配额、降轻任务时延 |
| Embedding / Rerank | 甲方部署 bge 系最新服务 | **前置验证**：embedding endpoint 须暴露 dense+sparse 双输出；版本钉死 + 变更提前通知（变更 = 全量重灌向量） |

MCP / SKILL：检索服务、cases SQL 工具（§6.6）以 MCP tool 形式注册到网关侧 ⚠（具体接入规范待甲方杨老师明确，列入 P0）。

### 9.2 多模型复核（非功能验收新增，CP-007 增强）

落位为**生成后校验步**而非全量双跑：主答（Qwen3.5）生成后，由 Kimi 2.5 对"回答中每条引用是否真实支持其论断"做 faithfulness 校验；不通过的句子降级"待核实"或触发重生成。

**CP-007 增强**：对 R5 判定型输出额外校验试探性表述（"可能违反""疑似违规""倾向于不合规"）是否被引用条款支持；不支持的强制改写为中性描述或降级"待核实"，**绝不踩"无依据结论"红线**。

全量双跑成本约 ×1.8 ⚠，仅在依据不足边界场景与 R5 判定型启用（其余路由不双跑）。

### 9.3 其余横切项

- **敏感词过滤**：网关入站/出站双向，词表对接甲方安全合规部门
- **AI 内容标识**：输出契约 `ai_label` 强制存在；R5 额外携带 `route_type=judgmental`，前端渲染"AI 辅助判断，人工复核"显式标记框；导出文件含固定标识页脚
- **权限**：Casbin RBAC+ABAC；检索前置过滤（§5.3）；菜单/数据/文件/导出/模板权限点 + 操作日志全量写 PG。启动会要求至少三级权限（普通员工/主管/总经理），由 perm_tag 与角色映射承载
- **观测**：Langfuse 全链路 trace（查询理解前端→检索→重排→生成→复核可回放）；HyDE 改写文本、事项分类标签、问题分解子查询、route_type 全部进 trace，供 V0 第 5 组 A/B 分析与查询分型统计（证伪监测）
- **SSO**：甲方统一认证对接，用户/组织同步只读

## 10. 输出契约与前端无关化

后端只交付统一 JSON 契约，genesis-ui 渲染只消费此契约（前端无关化，满足 Vue + 甲方组件库硬约束）。

```json
{
  "route_type": "evidence | change | case | enumerate | judgmental | statistical | clarify | refuse",
  "answer_blocks": [
    {"type": "text | table | case_card | clarify_question", "content": "...", "stream": true}
  ],
  "citations": [
    {"clause_id": "...", "doc_title": "...", "doc_no": "...",
     "clause_path": "第三章第二十一条第二款", "page_start": 7, "page_end": 7,
     "version": "...", "status": "effective | superseded | abolished"}
  ],
  "confidence": 0.0,
  "ai_label": true,
  "review_required": false,        // R5 判定型 = true，前端渲染人工复核框
  "exhausted_scope": ["投顾业务"], // 覆盖感知拒答时填，标"已穷尽事项分区"
  "export_enabled": true
}
```

要点：引用四级定位由后端从 PG 回查填充（§7.3）；`route_type` 驱动前端差异化渲染（判定型显式带人工复核框）；流式块通过 `stream:true` 标记增量推送。

## 13. 与语料评估 V0 的衔接（查询侧相关项）

V0 双轨（统计轨 + 实验轨，工期约 1.5 周 ⚠，复用 demo Docker 栈）中与查询智能体直接相关的：

- **统计轨 — 术语断层率** ⚠：从制度映射表/专家给出的 30–50 对已知"外规义务↔内规覆盖条款"抽样，统计词面无重叠且向量相似度低于召回阈值的"双低"配对占比。断层率 >20% ⚠ → 词典/桥接通道先验价值高；<5% → 混合检索 + 词典可能足够。双低配对直接构成 POC 盲评集的术语断层子集
- **实验轨 — 第 5 组 HyDE on/off A/B（CP-007）**：用 18 问术语断层样本 + V0 断层配对集作输入，对比口语问句改写为假设性法言后召回 hit@10 提升幅度。决策点见 §3.1。该实验直接决定 §12 中"查询理解前端开销"按 100% 还是按桶计入配额
- **实验轨 — 拒答阈值标定**：对抗子集标定 §8.1 充分性自检阈值与覆盖感知拒答的"事项分区穷尽"判据阈值
- **合成评估集**：200 件样本生成 300–500 问，含 ≥15% 多跳、≥15% 对抗/拒答、**≥10% 行为合规咨询型**（从 18 问 + 应用场景萃取扩增）⚠；经摄取期 T1 增殖至 W3 末 ≥2,000 问 ⚠，作为查询智能体永久回归资产
- **行为合规咨询型子集**：作为判定型路由（R5）的专项回归集，线上真实日志到位后逐步替换合成问题

评估指标：RAGAS（faithfulness / answer relevancy / context precision-recall，arXiv:2309.15217）+ 引用准确率盲评。KPI：context recall ≥0.95（非协商）、引用准确率硬地板 ≥95%。
