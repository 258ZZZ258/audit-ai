# 制度查询智能体 + 制度比对智能体：RAG 技术框架与工作流设计（V1.5）

> 适用项目：东方证券审计大模型系统一期
> 设计依据：项目启动会修订版文档（2026-06）+ 甲方提问方式与应用场景样本（2026-06）+ 既有架构决策（LangGraph / DeepDoc / Milvus / 必要性覆盖范式 / 六层防漏检）

> 文档性质声明：本文区分【行业标准/成熟实践】与【合成启发式（工程判断）】，后者均以 ⚠ 标注，须经张翼飞复核后方可对甲方陈述。

**一期边界（不做）**：不回写任何源系统；不自动修改制度正文；知识库更新为人工导入；不建设审批/复核流转。数据流严格单向只读。

**专业判断/概念辨析类任务不在本文档承诺范围（V1.5 边界声明）**：应用场景文档中的概念等价判断（如增发是否等于换股）、因果方向辨析（如估值-会计估计变更）、监管认定与审计要点合成等任务，需要多步检索-推理循环、推理链可追溯与推理有效性复核——其工作流形态（plan→retrieve→reason→re-retrieve→synthesize 循环）、产物形态（推理链而非引用定位）、复核机制（推理有效性而非引用真实性）均与"检索+引用约束生成"的查询智能体存在结构性差异。这类任务建议独立立项或挂载至场景 4（审计规范生成）扩展，**不在 V1.5 承诺范围**。本文档面向的"案例桥接检索"（§6.5 CP-007）属于查询增强通道,**不构成案例分析能力**。

## 1. 总体架构

延续"1 平台 + 多形态应用"原则。本文档涉及的两个智能体同属 **Chatbox 型**（制度查询）与 **后台任务型**（制度比对——尽管入口是页面交互，其核心是长耗时批量比对任务，按后台任务型设计）。

```
┌─────────────────────────────────────────────────────────────────┐
│ 业务应用层                                                        │
│  ┌──────────────────┐   ┌──────────────────────────────┐        │
│  │ 制度查询智能体      │   │ 制度比对智能体                  │        │
│  │ (Chatbox型/同步流式)│   │ (后台任务型/异步+状态轮询)        │        │
│  └────────┬─────────┘   └──────────────┬───────────────┘        │
├───────────┼─────────────────────────────┼────────────────────────┤
│ 平台能力层  ▼                             ▼                        │
│  ┌─────────────────────────────────────────────────────┐        │
│  │ RAG检索服务（混合检索/重排/版本过滤/权限过滤）              │        │
│  │ 查询理解前端（HyDE改写/业务事项分类/问题分解，CP-007）       │        │
│  │ Agent编排服务（LangGraph：8路意图路由/多轮收敛/工具调用）   │        │
│  │ 文档比对服务（义务抽取/覆盖判定/冲突判定/差异汇总）           │        │
│  │ 引用追溯服务（条款→文档→页码→版本 四级定位）                │        │
│  │ 情景术语桥接词典服务（口语↔法言映射，CP-007）              │        │
│  │ 报告/导出服务（Excel模板填充，复用模板库）                  │        │
│  └─────────────────────────────────────────────────────┘        │
├──────────────────────────────────────────────────────────────────┤
│ 基础设施层                                                         │
│  模型网关(OpenAI兼容,强制) │ Milvus 2.4 │ PostgreSQL │ Casbin     │
│  Temporal(异步任务) │ Langfuse(观测) │ MinIO/文件存储 │ 敏感词服务  │
├──────────────────────────────────────────────────────────────────┤
│ 数据接入层（单向只读，一期以文件导入为主，接口预留）                     │
│  内规库(1k) │ 外规+监管问答(20k) │ 监管案例(10k) │ 制度映射表 │ 修订说明 │
│  ※ 监督共享/费用/审计报告资料/模板的摄取边界见 §13（CP-008）            │
└──────────────────────────────────────────────────────────────────┘
```

两智能体**强制复用**同一套：解析管线、Chunk 仓库、Embedding 服务、Milvus 集合、引用追溯 schema、模型工厂。比对智能体不得另建私有检索通道（schema 变更遵循 add-only 约束）。

## 2. 语料分层与规模测算

### 2.1 三类语料的工程差异（决定差异化处理策略）

| 维度 | 内规 (1k) | 外规+监管问答 (20k) | 监管案例 (10k) |
|---|---|---|---|
| 在两智能体中的角色 | 查询主体之一；比对中的"覆盖方" | 查询主体之一；比对中的"义务来源方" | 查询的**桥接通道入口+附挂输出**（CP-007）；比对的反推校验输入 |
| 处理保真度 | **最高**：条款级结构化 + 全量人工抽检 | 高：条款级结构化 + 分层抽检 | 中：要素抽取为主（处罚对象/事由/依据条款/金额/日期） |
| 版本管理 | 完整版本链（修订说明、废止替代） | 完整版本链（新旧版 diff 是比对第一遍的输入） | 无版本概念，按发布日期管理 |
| 知识图谱纳入 | 是（全量 1k） | **仅映射子集**（见 §4.3） | 否（纯向量 + 结构化字段） |
| 元数据要求 | 发文字号/生效日期/业务域/密级/状态 | 发布机构/层级(法律→规章→自律规则→问答)/业务域/状态 | 处罚机构/对象类型/违规类别/引用外规条款 |

### 2.2 规模测算 ⚠（工程估算，须经语料评估 V0 实测修正）

| 假设项 | 内规 | 外规 | 案例 | 合计 |
|---|---|---|---|---|
| 文档数 | 1,000 | 20,000 | 10,000 | 31,000 |
| 平均篇幅（页）⚠ | 15 | 10 | 3 | — |
| 切块策略 | 条款级，目标 300–600 token | 同左 | 全文+要素，~400 token | — |
| 估算 chunk 数 ⚠ | ~4 万 | ~50–80 万 | ~8 万 | **~60–95 万** |
| bge-m3 dense 1024d fp32 体积 | 0.16 GB | 2–3.3 GB | 0.33 GB | **~4 GB** |
| Milvus 内存预算（HNSW，×1.5 系数）⚠ | — | — | — | **单节点 16–32 GB 可承载** |

**结论性判断**：百万级 chunk 在 Milvus 单机 standalone 即可承载，一期**不需要**分布式集群；按 corpus 类型做 partition key 分区即可。该结论须在语料评估 V0 用真实 200 件抽样的平均 chunk 数回归修正——若外规平均篇幅显著高于 10 页（例如大量法律全文），chunk 总量可能上探 150 万+，届时内存预算上调至 64 GB，仍不改变单机结论。

**Embedding 摄取耗时** ⚠（CP-005 修订）：60–95 万 chunk 经甲方部署的 bge 系向量化服务批量编码，纯计算预计小时级～半天（取决于该服务吞吐与配额，对接会确认）；属一次性离线任务，可接受。增量更新（制度月度新增）量级为每月数十件，分钟级完成。

## 3. 摄取管线（Ingestion Pipeline）

两智能体共用一条管线，**离线批量 + 增量导入**双模式。对应启动会"文件导入与解析"支撑工作及"文件解析风险"应对。详细设计见《文档处理与语料库构建_技术框架设计》V1.5。

```
文件导入(Word/PDF/Excel/扫描件)
  → ① 格式探测与预处理（扫描件走 PaddleOCR 分支）
  → ② DeepDoc 结构化解析（标题层级/条款编号/表格/页码锚点）
  → ③ 解析质检关卡：结构化得分 < 阈值 → 进入人工补录队列（带失败原因）
  → ④ 条款级切块（保留 章-节-条-款 路径 + 页码区间）
  → ⑤ 元数据标注（规则抽取为主 + LLM 辅助 + 人工确认敏感字段）
  → ⑥ 版本链登记（PostgreSQL：废止/替代/修订关系，人工录入修订说明）
  → ⑦ 双路写入：
       7a. Milvus（dense + sparse 双向量，bge-m3 一次编码产出）
       7b. PostgreSQL 条款表（全文 + 结构路径 + 引用锚点，支撑精确回显）
  → ⑧ （仅比对范围语料）LightRAG 图抽取队列 → 专家审核 → 图入库
```

**与功能 5/6/7/8/9 摄取的边界**：本管线服务知识库类语料（制度/外规/QA/案例）。监督共享、费用、审计报告资料、模板的摄取需求与本管线不一致，由 §13（CP-008）声明边界，详细设计另起文档。

关键决策点：

- **③ 解析质检是硬关卡**。启动会已将"文件解析风险"列为已知风险并要求"解析失败提示、人工补录"机制——管线必须把质检做成阻断点而非告警点：未过质检的文档不进入检索库，避免污染引用定位。质检指标 ⚠：条款编号识别率、页码锚点完整率、表格还原率，阈值在语料评估 V0 中标定。
- **⑤ 元数据中"状态"字段（现行/废止/即将生效）是检索默认过滤条件**，直接决定查询智能体不会引用废止制度——这是依据验收的隐含前提。
- **⑦b PostgreSQL 条款表是引用追溯的权威源**：向量库只存检索用切块，回答中展示的条款原文、页码、版本一律从 PG 回查，避免 chunk 截断导致引用失真。

## 4. 存储拓扑

### 4.1 Milvus（2.4，单机 standalone）

| 集合/分区设计 | 说明 |
|---|---|
| 单 collection `audit_corpus`，partition key = `corpus_type` (internal / external / case) | 统一 schema 便于跨库混合检索（查询智能体需要同时打内规+外规） |
| 向量字段：dense(1024, bge-m3) + sparse(bge-m3 lexical) | Milvus 2.4 原生支持 sparse 向量与 hybrid search + RRF 融合（成熟特性，非自研） |
| 标量字段：doc_id, clause_path, status, biz_domain, issuer_level, effective_date, perm_tag, **entity_type（CP-007 新增，适用实体类型如 C类营业部/A类分支机构/上市公司控股子公司，由 E2 打标维度同步扩展）** | status / perm_tag 为强制过滤位；biz_domain 支撑比对按业务圈定；entity_type 支撑 18 问中"C类营业部需要做机房出入登记吗"这类实体型问句的精确过滤 |
| 索引 ⚠ | dense 用 HNSW（M=16, efConstruction=200 起步）；百万级规模下无需 IVF 量化 |

### 4.2 PostgreSQL（权威结构化存储）

核心表（add-only 原则，预留权限字段）：

- `documents`：文档登记（含状态、密级、来源系统、导入批次）
- `doc_versions`：版本链（prev_version_id、修订说明全文、废止替代类型、发布通知引用）
- `clauses`：条款表（结构路径、原文、页码起止、所属版本）——引用追溯权威源
- `regulation_mapping`：制度映射表（外规条款 ↔ 内规文档/条款，多对多，含来源：自动抽取/专家录入/案例反推）
- `obligation_keywords`：义务关键词词典（六层防漏检第 2 层的硬过滤词表，专家可维护）
- `comparison_tasks` / `comparison_findings`：比对任务与差异结果（三分类 + 依据清单 + 判定置信度 + 复核状态）
- `cases`：案例要素表（处罚机构/对象/事由/引用外规条款/金额/日期）
- **`dict_scenario_terms`（CP-007 新增）**：情景术语桥接词典，由张老师及业务专家维护，dict_version 管理;映射示例:"代客理财"→"全权委托/受托理财"、"见底到顶"→"对买卖时机的具体建议"、"二维码介绍开户"→"违规招揽客户/居间介绍"。该词典服务 sparse 通道扩展与业务事项分类前置匹配,直接服务 18 问中的术语断层问句
- **`dict_intent_routes`（CP-007 新增）**：意图路由训练样例库,含真实查询日志样本与人工标注的 §5.7 八路意图标签,供路由器持续优化

### 4.3 知识图谱：**按需懒构建，范围收敛**

20k 外规全量建图不可行（LLM 抽取调用量级 ≈ 数十万次，成本与专家审核工时均不可接受 ⚠）。锁定策略：

1. 图谱范围 = 内规 1k 全量 + **制度映射表命中的外规子集**（估算 2–5k 件 ⚠，待映射表确认后修正）
2. 构建时机 = **比对任务驱动的懒构建**：某业务域首次发起比对时，触发该域映射外规的图抽取队列；抽取结果经专家"批准/驳回/修改"后入库（沿用既定的 LLM 候选 + 专家审核模式）。队列内部按 `graph_priority_score`（摄取文档 §19.3 E3 探针）排序，专家人时优先投向图谱收益最高的文档（CP-002）
3. 本体上限不变：5 类节点 + 6 类关系。**抽取输入粒度按 CP-001 解耦**：输入为条款树装配的节级窗口（含确定性指代预解析注释），非 chunk 直喂；三元组强制条款级锚定与窗口成员校验——POC 的"引用准确率 ≥95%"明确为锚点校验通过率（摄取文档 §6.6/§6.7）
4. 存储：NetworkX pickle（demo）→ Neo4j profile（生产可选），与制度查询智能体既有默认一致
5. （CP-003 措辞修正）制度查询智能体一期**不配知识图谱通道**——其检索实为混合检索（dense+sparse）+ 结构化通道（版本链 / clause_references 指代表 / 案例要素反查 / 标量过滤）+ **CP-007 新增的查询理解前端与案例桥接通道**。不配图谱是带可证伪条件的工程押注:依据为查询形态分布假设 ⚠(单点定位为主,多跳已被指代表/版本链/问题分解节点确定性拦截)与图谱供给不足。**证伪监测**:依据不足事件全量进 Langfuse + 真实查询日志分型统计,多跳关联类占比 >15% ⚠ 或依据不足聚类集中于关联类问题时触发二期接入评估;**回退成本**:LangGraph 加一个检索节点 + 路由分支,特征已由 doc_graph_stats 预留(E4),存储与引用 schema 零改动。4 周图谱 POC 的通过阈值不变(引用准确率 ≥95%、关系校准 <3min、盲评提升 ≥0.5 分)。

## 5. RAG 核心管线（全链路口径）

按既定的全管线分析结构逐环节给出选型与理由。【标准】= 成熟实践；⚠ = 工程判断。

| 环节 | 选型 | 理由 / 备注 |
|---|---|---|
| 5.1 解析 Parsing | RAGFlow DeepDoc 主通道；扫描件 PaddleOCR；复杂版式兜底 MinerU | 【标准】制度类文档以规整版式为主，DeepDoc 对中文条款结构识别成熟 |
| 5.2 切块 Chunking | 条款级结构化切块（按 条/款 边界），300–600 token，父子块（child=条款，parent=节）⚠ | 制度文本天然有条款边界，结构切块优于固定窗口（NVIDIA 2024 chunking benchmark 支持结构感知切块结论）；父子块用于"小块检索、大块供证"的引用完整性 |
| 5.3 嵌入 Embedding | bge-m3（dense+sparse 一次产出），API-first 推理 | 既定默认；中文法规域表现稳定，sparse 输出免去单独 BM25 服务 |
| 5.4 向量库 | Milvus 2.4 单机，partition key 分区，标量强过滤 | §4.1 |
| 5.5 检索 Retrieval | Hybrid（dense + sparse）+ Milvus RRF 融合；默认 status=现行 + 权限过滤前置（filter 而非后过滤） | 【标准】混合检索对法规类专有名词（发文字号、罚则编号）的精确命中显著优于纯稠密；权限过滤必须在检索阶段完成 |
| 5.6 重排 Reranking | bge-reranker-v2-m3，top50 → top8 ⚠ | 既定默认；top-k 数值待语料评估标定 |
| 5.7 查询理解前端 + 路由（CP-007 升级）| **查询理解前端三节点**:① HyDE 改写(口语 → 假设性法言条款,跨术语断层)、② 业务事项分类(映射 E2 涉及事项标签 + dict_scenario_terms 桥接 + entity_type 抽取)、③ 问题分解(复合问句拆为子查询;单跳直通)。**LangGraph 路由 8 路**:依据查询 / 变更查询 / 相似案例 / 多文档列举(含枚举模式)/ **判定型(§6.8)** / **统计型(§6.9)** / 多轮澄清 / 拒答兜底 ⚠ | §6.1/§6.2/§6.8/§6.9。HyDE 默认 on/off 待 V0 实验轨第 5 组 A/B 实测(§10.2) |
| 5.8 生成 Generation | Qwen3.5（甲方网关满血模型，CP-005）主答；引用强约束 prompt（只允许引用检索上下文中带 clause_id 的内容）；流式输出 | 引用 ID 由系统注入，LLM 仅做选择不做生成，从机制上杜绝编造发文字号 |
| 5.9 输出契约 | 统一 JSON：answer_blocks[] + citations[]（clause_id→doc→page→version→status）+ confidence + ai_label + **route_type（CP-007,标记路由分支供前端差异化渲染:判定型显式带"AI 辅助判断,人工复核"框)** | 前端无关化：genesis-ui 渲染只消费此契约，引用四级定位由后端从 PG 回查填充 |
| 5.10 评估 Evaluation | RAGAS 指标（faithfulness / answer relevancy / context precision-recall）+ V0 合成评估集（300–500 问，含多跳/拒答子集，永久回归资产）+ 引用准确率盲评 | 【标准】RAGAS (arXiv:2309.15217) + 合成基准范式（DataMorgana / RAGAS testset）；详见 §10（CP-004 + CP-007 第 5 组 A/B） |

## 6. 制度查询智能体工作流（Chatbox 型，同步流式）

### 6.1 LangGraph 状态机（V1.5 CP-007 升级）

```
[用户输入] → 多轮上下文归并(指代消解/补全省略的制度名)
   → 查询理解前端(CP-007):
      ① HyDE 改写(口语 → 假设性法言)         [HyDE 节点,默认 on,边界场景 off 待 V0 标定]
      ② 业务事项分类(E2 事项 + dict_scenario_terms 桥接 + entity_type 抽取)
      ③ 问题分解(复合问句拆为子查询;单跳直通)
   → 意图路由(8路)
   ├─ 依据查询 ──→ 混合检索(内规+外规分区并行)∥案例桥接通道(§6.5 升级)
   │               → 重排 → 版本/权限/entity_type 过滤 → 充分性自检
   │               ├─ 充分 → 引用约束生成(流式) → 案例附挂 → 输出+导出按钮
   │               └─ 不足 → 覆盖感知拒答(§6.6 升级)
   ├─ 变更查询 ──→ 制度定位 → 版本链回查(PG) → 条款级 diff + 修订说明/背景归并
   ├─ 相似案例 ──→ 案例 partition 检索 → 要素表回填 → 案例卡片输出
   ├─ 多文档列举 ─→ E1 义务标签 ∩ E2 事项标签 ∩ biz_domain ∩ entity_type 过滤
   │                (枚举模式:高 k 不激进截断,确保召回完整性,CP-007)
   ├─ 判定型(§6.8) → 行为合规咨询型问句 → 案例桥接为主 + 依据条款 + 构成要件框定
   │                  → 多模型复核(校验试探性表述可支持性) → 三段式"依据+框定+人工复核"输出
   ├─ 统计型(§6.9) → 维度抽取 → cases 参数化 SQL 聚合 → 表格化输出
   ├─ 需澄清 ───→ 单问题澄清(纯对话,不出复选框)
   └─ 兜底 ─────→ 礼貌拒答(超出审计制度域)
```

### 6.2 路由设计要点（CP-007 升级）

- **HyDE 改写**:由网关轻量模型(CP-005-①待确认,无则满血)生成 1-2 句假设性法言条款,与原始问句一同送入检索的 sparse/dense 通道;失败回落原始问句。HyDE 编出貌似合理的错误法言不污染最终答案,因引用 ID 注入(§5.8)兜住。HyDE on/off 进 V0 实验轨第 5 组 A/B(§10.2),决定默认开启还是术语断层桶触发。
- **业务事项分类器**:轻量分类模型,标签空间 = E2 涉及事项字典 ∪ 用户提问场景类型(行为合规咨询 / 概念定义 / 列举枚举 / 变更追溯 / 统计聚合)。问句的"场景类型"标签直接驱动路由分支选择。
- **问题分解节点**:仅对显式复合问句(如"私募权益投顾同时管偏股+偏债是否违规"含多个子约束)触发,拆为子查询后并行检索再综合。单跳问句直通,不引入额外延迟。**不进入 plan→retrieve→reason→re-retrieve 循环**(参见 §0 边界声明)。
- 路由器训练样例库 `dict_intent_routes` 须含金总提供的真实查询日志——这是日志数据的第一个落点,且提问方式 18 问 + 应用场景 5 问应直接灌入作为冷启动种子。
- 多轮收敛遵循硬约束:**纯 chatbox,不出预设勾选条件**(甲方明确否决过复选框)。

### 6.3 依据查询：检索细节

- 内规与外规**并行检索各自 partition**，各取 top25 重排合并 ⚠——避免外规 20 倍体量在统一检索中淹没内规结果（分区配额是对语料不均衡的直接对策）
- 发文字号、制度全名走 sparse 精确通道权重提升；语义问句走 dense 主导
- 默认 status=现行；用户显式问历史版本时放开过滤并在回答中标注"该版本已废止/被替代"
- **entity_type 过滤(CP-007)**:问句若被业务事项分类器识别出实体类型(如"C类营业部"),检索阶段强制过滤 entity_type 标量,避免召回适用于其他类型实体的条款

### 6.4 变更查询：版本链 + 修订说明

启动会要求输出"版本、变更内容、**变更原因、背景情况**"。变更内容可由条款级 diff 机器产出；**变更原因和背景无法从正文 diff 推断**，权威来源是修订说明和发布通知。

工程处理：
1. `doc_versions` 表存修订说明全文，摄取时按"修订条目"切分并与 diff 出的条款变更做对齐（LLM 辅助对齐 + 置信度标注）⚠
2. 修订说明缺失时：变更内容照常输出，变更原因栏明确显示"修订说明未提供，无法给出官方修订原因"——**不允许 LLM 推测原因**
3. 背景情况允许附挂同期监管动态案例（来自案例库，按日期窗口检索），但必须标注为"同期监管背景参考"而非官方原因

### 6.5 案例桥接检索（V1.5 CP-007 升级，从"附挂"扩为"桥接通道+附挂"双通道）

**核心洞察**:处罚案例正文用接近口语的事实描述("当事人通过微信发送开户推广二维码……"),同时 `cases.引用外规条款` 字段已结构化指向所违反的外规条款。对**行为合规咨询型问句**(用户用具体业务场景的口语描述、命中的却是法言法语条款,如"二维码介绍开户违规吗""见底到顶算不算买卖建议"),案例正文与口语问句词面高度重合,可作为"口语 ↔ 法言"的术语桥梁——案例从 V1.4 的"事后装饰"升级为"检索路径本身"。

**双通道并存**:

| 通道 | 用途 | 工作流位置 |
|---|---|---|
| 桥接通道(CP-007 新增) | 行为合规咨询型问句的检索入口:口语问句先打 case partition → 命中案例 → 顺 `cases.引用外规条款` 精确跳转至外规条款 → 拿到被违反的外规条款原文并送入主答上下文 | §6.1 依据查询/判定型主路径前置(与混合检索并行) |
| 附挂通道(原 V1.4 设计) | 依据查询回答完成后追加:精确反查(外规条款 → cases)∪ 语义检索 case partition,取 top3 卡片附在回答尾部 | §6.1 依据查询主路径后置 |

**触发条件(仅桥接通道)**:业务事项分类节点(§6.2)判定问句类型为"行为合规/违规咨询型"时触发;定义型/列举型/变更型/统计型问句不走桥接通道(避免过度推销)。

**适用边界(明确不做)**:
- 概念判断型问句("增发是否等于换股"):无对口案例,不走桥接
- 因果辨析型问句("估值与会计估计变更因果方向"):超出查询智能体范围,参见 §0 边界声明
- 纯依据查询("费用报销发票3个月规定在哪"):精确依据已足,无需案例中介

**与比对智能体第 6 层的关系**:本通道与 §7.3 第 6 层"案例反推补充检测"复用同一 `cases.引用外规条款` 精确反查机制,无新增工程开销,仅查询侧消费路径不同。

### 6.6 覆盖感知拒答与判定型框定（V1.5 CP-007 升级）

V1.4 的依据不足策略基于"top1 分数阈值"判断——对**存在性问句**("有没有规定…""违不违规""需不需要")存在系统性失效:检索为空时,系统区分不了"确实无禁止性规定"和"我没召回到",输出"依据不足"会让甲方误以为系统无能。CP-007 升级判据:

**1. 分数阈值 → 覆盖语境判据**

判据由"top1 分数 < 阈值"升级为"是否在**相关业务事项分区内穷尽检索过**":

- 问句经业务事项分类(§6.2)映射到 E2 涉及事项(如 `事项=投顾业务`)
- 在该事项分区内执行高召回检索(top 50,所有候选 E1 义务条款)
- 仍无命中 → 输出**覆盖感知拒答话术**(替代旧的"依据不足"):

> "在【投顾业务】现行制度中**未检索到对该行为的明确禁止性规定**,但不等于无规定(可能涉及其他事项分类或穿插条款)。以下 N 条最接近供人工核实:……"

**2. 判定型框定(防止"无依据结论"红线踩线)**

判定型路由(§6.8)的输出形态:

- **绝不出"违规/合规"裸结论**——红线
- 输出三段式:① 相关依据条款(带四级锚点)+ ② 构成要件框定("该条规则的适用前提为 X、Y、Z;本问句涉及的事实需具备…")+ ③ "AI 辅助判断,建议人工复核"标识
- 多模型复核(§8.2)专门校验任何试探性表述("可能违反""疑似违规")是否被引用条款支持;不支持的表述降级为"待核实"

**3. 与红线的关系**

| 红线 | V1.4 旧策略风险 | V1.5 升级 |
|---|---|---|
| 不生成无依据结论 | 直接给"违规/合规"判定 → 踩线 | 判定型路由仅输出"依据+构成要件",不出裸结论 |
| 拒答应可解释 | "依据不足" 无信息量 | 覆盖感知拒答附"已穷尽的事项分区+最接近的 N 条" |
| 引用真实性 | 多模型复核覆盖引用真实性 | CP-007 升级:复核同步检查"试探性表述是否被引用支持" |

**待确认**:判定型路由"不出裸结论"的产品形态需经张益确认是否满足业务期望(§11-10)。

### 6.7 导出

Excel 导出复用报告/导出服务 + 模板库（模板库见 §13.5）:查询结果模板含 问题 / 答复摘要 / 依据条款（四级定位）/ 相似案例 / 路由类型 / 导出人 / 导出时间 / AI 内容标识。导出动作过 Casbin 导出权限点并写操作日志。

### 6.8 判定型路由（V1.5 CP-007 新增）

**触发**:业务事项分类(§6.2)判定为"行为合规/违规咨询型"问句(甲方提问方式 18 问中此类占比约 1/3,含"二维码介绍开户""见底到顶算不算买卖建议""专用章保管""机房出入登记"等)。

**工作流**:

```
判定型问句
 → 桥接通道(§6.5 升级):案例反查 → 外规条款定位
 → 混合检索(内规+外规)补充候选条款
 → 重排聚合 → 构成要件提取(LLM 从命中条款抽取"该规则的适用前提/对象/行为类型")
 → 三段式输出生成(prompt 模板硬约束输出结构):
    ① 相关依据条款(四级锚点)
    ② 构成要件框定(结构化展示该条款的适用边界)
    ③ "AI 辅助判断,建议人工复核"标识 + 多模型复核标记
 → 引用约束 prompt 强制"不出违规/合规裸结论"
 → 多模型复核校验试探性表述(§8.2)
 → 输出(route_type=judgmental,前端差异化渲染人工复核框)
```

**为什么不走依据查询路由**:依据查询的产品形态是"问什么答什么",judgmental 问句若走该路由,主答模型容易顺势给出裸结论(即使引用条款是真的)。判定型路由通过 prompt 模板硬约束输出结构,从形态上杜绝裸结论。

**关键设计点**:

- **构成要件框定**而非"我也不知道":案例桥接已经把最接近的依据捞到,构成要件框定是把"判断这一步"留给人——这是工程能做的极限,也是红线允许的边界。
- **不引入多步推理循环**:判定型路由是单轮检索+结构化输出,不进入 plan→retrieve→reason→re-retrieve→synthesize 的 agentic 循环(参见 §0 边界声明,案例分析能力不在 V1.5 范围)。

**与比对智能体的边界**:比对智能体的"差错点"判定(§7.4)是有明确依据(双向交叉验证或硬规则触发)的差异认定;判定型路由处理的是"用户不知道规则、想知道行为是否触线"的咨询场景,无既有结论,二者不重复。

### 6.9 统计型路由（V1.5 CP-007 新增）

**触发**:业务事项分类判定为"聚合统计型"问句(如"哪些板块是处罚高发的板块""2024年以来期货监管处罚有哪些")。

**工作流**:

```
统计型问句
 → 维度抽取(LLM 从问句识别聚合维度:板块/事由/时间窗/处罚机构/对象类型等)
 → 参数化 SQL 生成(基于 `cases` 表结构化字段:违规事由分类、处罚机构、
   处罚日期、对象类型、处罚金额等;SQL 模板预定义,LLM 仅填参数,防注入)
 → SQL 执行 → 结果表格化输出(按聚合值排序)
 → 可选:附带 top N 案例链接供下钻查看
```

**关键设计点**:

- **不走向量检索**:统计聚合类问题用 top-k 给不出答案,必须走 SQL。
- **前提依赖**:cases 表的 §9 字段完整率 ≥90% 闸要守住,违规类别字典(§16-6)需经张老师评审落地。
- **与场景 5 边界**:场景 5(监管案例舆情分析)是后台任务型报告,周期性生成完整分析报告;统计型路由是查询智能体内的轻量分析通道,即问即答,适合临时性聚合查询。二者数据源同为 cases 表,无重复。

**输出形态示例**:

| 问句 | 输出 |
|---|---|
| "2024年以来期货监管处罚有哪些" | 按发布日期降序的案例列表(标题+处罚机构+处罚日期+处罚类型) |
| "哪些板块处罚高发" | 按违规事由分类聚合的板块计数表,降序排列 |

## 7. 制度比对智能体工作流（后台任务型，异步）

### 7.1 任务入口与形态

页面输入：被审计单位 / 指定业务 / 指定制度（三种圈定方式可组合）。提交后创建 Temporal 工作流，前端轮询任务状态（排队/义务抽取中/覆盖判定中 x/y/汇总中/完成/部分失败），契合"长耗时任务异步处理 + 状态展示"的性能验收要求。

### 7.2 两遍法总流程

```
任务创建(圈定: 单位/业务/制度)
 → Pass 1: 外规新旧版差异
     映射表圈定外规集合 → 各外规取现行版 vs 上一版 → 条款级diff
     → 新增/修订的"应当/必须"类义务条款 → 进入Pass 2优先队列
 → Pass 2: 内规对外规的必要性覆盖
     ① 义务条款抽取(外规侧):读取摄取期 E1 预打标(is_obligation/deontic_type/数值期限结构化列,
        摄取文档 §19.1) + 现场抽取兜底(防词典盲区) —— CP-002
     ② 对每条义务: 三路并行召回(内规侧)
     ③ 双模型集成判定 → 四态分类
     ④ 硬规则通道(数字/期限/比例)
     ⑤ 案例反推补充检测
 → 差异汇总 → 处理建议生成 → 依据清单装配 → 人工复核标记 → Excel导出
```

### 7.3 六层防漏检在管线中的落位

| 层 | 机制 | 管线位置 |
|---|---|---|
| 1 | 双模型集成（Qwen3.5 + DeepSeek V4 Pro，结果取**并集**，CP-005） | ③ 判定 |
| 2 | 义务关键词词典硬过滤（"应当/必须/不得/禁止" + 业务专家词表，PG 可维护） | ① 抽取（漏抽兜底）+ ② 召回（关键词通道） |
| 3 | 三路并行召回：dense（召回偏置，top50 低阈值）∥ sparse/关键词 ∥ 图谱路径（映射表已建图部分） | ② 召回 |
| 4 | 严格模式 prompt + 双向交叉验证（外规→内规判覆盖；内规→外规判依据，二者矛盾即标"待复核"） | ③ 判定 |
| 5 | 数字/期限/比例字段走硬规则比对，**不交给 LLM 判断**（如"5 个工作日 vs 10 个工作日"由规则引擎判差错） | ④ 硬规则 |
| 6 | 监管处罚案例反推：案例库中引用了圈定外规条款的处罚事由 → 反查内规是否有对应控制条款 → 命中空白即生成"案例提示型缺失候选" | ⑤ 案例反推 |

> 第 6 层注：CP-007 升级中 `cases.引用外规条款` 字段在查询智能体侧已成检索入口(§6.5 桥接通道),与比对第 6 层共用同一精确反查机制,无新增工程开销。

### 7.4 四态分类（启动会三分类 + 已覆盖）

| 分类 | 判定逻辑 | 主要责任通道 |
|---|---|---|
| 已覆盖 | 双模型均判"内规条款充分覆盖外规义务" | LLM 集成 |
| 部分覆盖 | 任一模型判"覆盖但范围/条件/对象不全"，附缺口描述 | LLM 集成 |
| **缺失点** | 三路召回均无候选，或双模型均判"无对应条款" | 召回 + LLM 并集 |
| **差错点**（启动会新增维度） | (a) 硬规则通道发现数值/期限/比例冲突；(b) 双向交叉验证发现内规要求与外规要求**方向相反或互斥** | 硬规则为主，LLM 为辅 |

差错点与必要性覆盖范式的关系:覆盖范式回答"内规有没有接住外规义务",差错点回答"接住了但接错了"。二者是叠加通道而非替代。

### 7.5 任务量与耗时测算 ⚠

单次典型比对（1 个业务域：内规 ~30 件 vs 映射外规 ~80 件）：
- 外规义务条款估算 800–2,000 条
- 每条义务：3 路召回（毫秒级）+ 2 模型判定 + 部分双向验证 ≈ 2.5 次 LLM 调用均摊
- 总 LLM 调用 ≈ 2,000–5,000 次 → 满血模型并发 8–16 路假设下 **30–120 分钟**（实际并发取决于网关配额，待对接会确认 CP-005-③；配额减半则耗时近似翻倍）
- Temporal 按义务条目分片，可断点续跑、可部分失败重试；任务状态以"已判定 x / 总数 y"粒度上报前端

### 7.6 输出与人工复核

产出物对齐启动会建设口径:制度清单、缺失点、部分覆盖、差错点、差异汇总、处理建议、依据清单。每条 finding 携带:判定分类、置信度、命中通道(哪几层防漏检触发)、外规条款四级定位、内规条款四级定位(缺失点则为空+检索过程留痕)、双模型判定是否一致。**双模型不一致或交叉验证矛盾的条目强制标"待人工复核"**,处理建议明确标注 AI 生成标识。比对结果只读展示与导出,不回写任何制度系统。

### 8.2 多模型复核（启动会非功能验收新增项，V1.5 CP-007 增强）

落位为**生成后校验步**而非全量双跑:主答(Qwen3.5)生成后,由 Kimi 2.5 对"回答中每条引用是否真实支持其论断"做 faithfulness 校验;校验不通过的句子降级为"待核实"标注或触发重生成。

**V1.5 CP-007 增强**:对判定型路由(§6.8)的输出额外校验试探性表述("可能违反""疑似违规""倾向于不合规"等)是否被引用条款支持;不支持的试探性表述强制改写为中性描述或降级"待核实",**绝不允许踩"无依据结论"红线**。

比对场景天然满足复核(双模型集成即复核)。全量双跑成本约 ×1.8 ⚠,仅在依据不足边界场景与判定型路由启用。

- **SSO**:甲方统一认证对接,用户/组织同步只读

## 10. 与语料评估 V0 的衔接（CP-004 + CP-007 第 5 组 A/B）

V0 角色重定义:原"语料统计 → 推断选型"模式中,多数选型已被锁定决策、运行期闭环(CP-003 可证伪监测)与直接实验(POC)接管。V0 收敛为双轨,工期 1 周 → 约 1.5 周 ⚠(实验环境复用 demo Docker 栈)。

### 10.1 统计轨（保留 / 删除 / 定向改造）

- **保留**:结构维度(外规篇幅分布 → chunk 总量与 Milvus 预算是否上修;条款覆盖、表格密度 → 质检阈值与正则标定)、工程维度(扫描件占比、可抽取性)、治理维度(密级/时效分布)、制度映射表覆盖率(图谱懒构建实际范围与比对圈定方式)
- **删除**:语义维度的 embedding/rerank **选型**职能——CP-005 后选型权在甲方(bge 系钉死),仅保留长度方差/术语分布统计用于切块参数与评估集分层
- **定向改造(关系维度)**:删去三元组密度等泛指标;新增**术语断层率** ⚠——从制度映射表或专家给出的 30–50 对已知"外规义务 ↔ 内规覆盖条款"抽样,统计词面无重叠且向量相似度低于召回阈值的"双低"配对占比。断层率 >20% ⚠ → 图谱/词典通道先验价值高;<5% → 混合检索 + 词典可能足够,POC 可缩水。双低配对直接构成 POC 盲评集的术语断层子集

### 10.2 实验轨（CP-007 新增第 5 组）

1. **合成评估集**:200 件样本生成 300–500 问,含 ≥15% 多跳、≥15% 对抗/拒答,**新增 ≥10% 行为合规咨询型(从甲方 18 问 + 应用场景萃取的种子模式扩增)** ⚠
2. **四组 A/B 当场实测**:混合 vs 纯 dense、面包屑前缀 on/off、父子块供证 on/off、重排 top-k 扫描
3. **拒答阈值标定**:对抗子集直接标定 §6.6 充分性自检阈值与分桶分数校准分位数;**CP-007 新增**:覆盖感知拒答的"事项分区穷尽"判据阈值标定
4. **图谱 POC 三臂对照**:A = 混合检索基线;B = 预建轻量图 + 专家审核;C = LazyGraphRAG 式查询时构建(模式重实现,非引入 graphrag 库)。量化四指标:漏检率 / 误报率 / 单义务 LLM 调用数 / 域级专家小时
5. **(V1.5 CP-007 新增)HyDE 改写 on/off A/B**:用 18 问中术语断层样本 + V0 断层配对集作为评估输入,对比口语问句改写为假设性法言后召回的 hit@10 提升幅度。决策点:
   - hit@10 整体提升 >5pp ⚠ + 边界场景无显著负面 → HyDE 默认 on
   - 仅术语断层桶有提升 → HyDE 仅在分类器识别为"行为咨询型 + 高断层"时触发(配额减半预案)
   - 无显著提升甚至污染 → HyDE 砍掉,回退混合检索 + 情景词典
   该实验直接决定 §9 容量测算中"查询理解前端开销"是按 100% 触发还是按桶触发计入网关配额申请

### 10.3 产出资产化

V0 产出 = 报告(指标→决策→建议 + 实验→实测结论,双方会签)+ **两件永久回归资产**:合成评估集、术语断层配对集。**V1.5 CP-007 增量**:行为合规咨询型问句子集(基于 18 问/应用场景 + 持续真实查询日志扩增)作为判定型路由的专项回归集。线上真实查询日志到位后逐步替换合成问题、校正分布偏差。评估集在 W1–W3 经摄取期 T1 组件持续增殖(V0 的 300–500 问 → W3 末 ≥2,000 问 ⚠),拒答对抗子集由 T5 负样本供给(摄取文档 §21,CP-006)。

- **本文档须经张翼飞复核(尤其 §6.5/§6.6/§6.8 判定型路由对"不出裸结论"红线的落实、§13 一期数据摄取全景的边界声明)后,方可作为对甲方汇报材料的底稿。**

## 13. 一期数据摄取全景与跨场景边界（V1.5 CP-008 新增，已批准）

### 13.1 背景

启动会修订版明确一期八项功能的覆盖范围,其中制度查询智能体(功能 1)、制度比对智能体(功能 2)、监管案例舆情分析(功能 3)的语料摄取由《文档处理与语料库构建_技术框架设计》V1.5 完整覆盖(P-INT/P-EXT/P-QA/P-CASE 四个 processing profile)。但**功能 5(监督共享)、功能 6/7(费用审计)、功能 8(审计报告资料)、功能 9(模板库)的数据摄取需求与现 pipeline 的处理形态不一致**,必须在原型阶段(M3 之前)至少有一版最小设计,否则 W2/W3 灰度时撞墙。本节给出边界声明 + profile 建议,**详细设计另起单独文档**。

### 13.2 功能 5：监督共享信息（建议 P-MISC profile）

启动会界定:合规、风险、审计部门月度固定导入"合规事项、问责情况、监管函件、法律诉讼、风险事项"等部门输出文档。

| 维度 | 与 P-INT/P-EXT 的差异 | P-MISC 建议 |
|---|---|---|
| 文档结构 | 无条款结构(报告/清单/通报为主) | 切块按段落/小节,无条款树构建;摄取文档 §5.1 质检指标 1(条款覆盖率)禁用 |
| 元数据 | 无发文字号、无生效日期 | manifest 必填强化为:来源部门 / 报送月份 / 文档类型(合规事项/问责/监管函件/法律诉讼/风险事项)/ 标签数组 |
| 版本链 | 无版本概念,按报送月份管理 | 月度批次 = 不可变快照,无 revise_replace 关系 |
| 引用追溯 | 不可达条款级 | 仅文档级+段落级,依据展示降级标识"非制度类来源" |
| 在哪个场景被消费 | 功能 5 的报告生成 | 与制度查询/比对智能体的检索范围**默认隔离**(不进 audit_corpus 主检索通道,单独 partition `corpus_type=supervision_share`) |

**关键边界**:P-MISC 文档**不参与制度查询智能体的依据查询路由**。功能 5 的报告生成有独立工作流,从 P-MISC partition 拉取,按时间维度汇总,不通过查询智能体路由。

### 13.3 功能 6/7：费用审计数据（混合摄取通道）

启动会界定:费用、预算、审批、发票、附件来自费控/预算/财务/审批等多系统,一期"接口预留+文件导入兜底"。

| 数据类型 | 现 pipeline 适配性 | 建议 |
|---|---|---|
| 发票图像 / 附件 PDF | 适配(OCR 通道,case partition 类比) | 独立 partition `corpus_type=expense_doc`;OCR 重点字段为金额/日期/对方主体 |
| 费用明细 / 预算 / 审批流水 | **不适配**(结构化业务数据,无文档形态) | 独立**业务数据库表族**(非 chunks 表,schema 单独设计:expense_records / budget_lines / approval_records);通过摄取文档 §4.1 xlsx 直读路径接入;查询走 SQL 而非向量检索 |
| 报销标准(属内规) | 适配(P-INT) | 走现 pipeline,作为费用审计的规则参考来源 |

**关键边界**:费用审计的"异常识别"主要依赖规则引擎对业务表的 SQL 查询,LLM 仅在"OCR 字段补全 + 违规原因解释"环节介入。这与查询智能体的"检索+生成"路径独立。

### 13.4 功能 8：审计报告项目资料（项目级生命周期）

启动会界定:被审计单位资料上传库 → 业务分析 → 三类报告生成 → 报告审核 → 版本留痕。

| 维度 | 与知识库语料的差异 | 建议 |
|---|---|---|
| 生命周期 | 项目期一次性使用(项目结束归档/删除) | 引入 project_id 维度,partition `corpus_type=audit_project`,按 project_id 分组;项目结束触发归档动作 |
| 检索范围 | 仅当前项目内 | 强制 perm_tag 过滤 + project_id 过滤;不与知识库主检索通道混合 |
| 与制度查询智能体的关系 | 项目资料是"被审计对象的数据",不是"判断依据" | 项目资料**不进入依据查询**结果;审计报告生成时与制度知识库联合检索(项目资料作事实素材,制度作判断依据) |

### 13.5 功能 9：模板库

启动会界定:Word/Excel/PDF 模板统一管理,含字段映射、章节配置、版本、默认模板、权限和使用留痕。

**关键边界**:**模板不属于语料**。模板是带占位符的可执行结构,与 chunks/cases 表族无关。建议独立子系统设计:

- 存储:MinIO(模板原件)+ PG(模板元数据:template_id / 名称 / 适用场景 / 字段映射 schema / 章节配置 / version / 默认标记 / 权限点 / 使用日志)
- 调用:场景 4(审计规范生成)、场景 8(审计报告生成)、查询智能体 §6.7 导出通过模板服务调用,传入 placeholder values → 模板服务渲染 → 输出 docx/xlsx
- 不进入查询智能体的任何检索路径
